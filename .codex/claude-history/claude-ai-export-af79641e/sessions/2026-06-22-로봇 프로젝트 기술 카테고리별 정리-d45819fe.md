기타 타브 채보 프로젝트는 STM32 ADC를 활용한 접촉감지와 아날로그 신호 처리 회로(증폭, HPF, OP앰프)를 FPCB에 구현하는 하드웨어 설계 중심이고, 30kV DC/DC 컨버터는 고전압 전력전자 설계와 PCB 몰딩을 포함한 실증 프로젝트다.

소 절단기는 BLDC 모터와 드라이버, 무한궤도 기구를 통합한 시스템이고, EV 컨버전 프로젝트는 20kW 3상 인버터와 CAN 통신을 연동한 대규모 전력전자 시스템으로 실차 통합까지 진행 중이다.

벽면 등반 로봇은 흡입 방식의 기구 설계와 실증에 집중하고 있으며, 출입 모니터링 시스템은 YOLOv3 기반 비전 처리와 함께 전원부 PCB 설계를 포함하고 있다.

화물 보조 모터드라이브는 ATmega128 마이크로컨트롤러로 전류와 토크를 제어하는 모터 드라이브 기술을 구현하고 있고, AI 자율주행 경로계획은 딥러닝 기반 인지와 경로 추종을 시뮬레이션으로 검증하는 중이지만 구체적인 시뮬레이션 환경이 명확하지 않으며 2026년까지 진행 예정이다.

자율주행 경로계획 프로젝트는 이미지가 없고 카테고리 매핑이 모호해서 6개 카테고리에 강제로 포함시키기 어렵다고 판단되어 일단 보류하기로 했다.

쿨링팬 자가진단도 마찬가지로 2026년 진행 중이고 이미지가 없으며, AI 진단 기능이 있지만 카테고리 6에 약하게 포함시킬지 보류할지 결정 중이다.

정리해보니 카테고리 3(Gazebo·Isaac Sim 시뮬레이션)에 명확히 해당하는 건 농기구추종 프로젝트뿐이고, 나머지는 도구가 불명확하거나 Isaac Sim을 명시한 사례가 없다.

카테고리 4(강화학습/Sim2Real)는 로보컵 2족과 안내견 4족 프로젝트가 있고, 로커보기는 JAX 최적화라 강화학습이 아니다.

조교님이 제시한 6개 카테고리가 실제 데이터와 완벽하게 매칭되지 않는다. 특히 ROS2와 Gazebo/Isaac Sim 카테고리는 프로젝트 수가 적다. 이를 정직하게 반영해야 하고, 데이터에 없는 내용을 억지로 채우면 안 된다.

내 접근은 6개 카테고리 각각에 데이터 근거가 있는 프로젝트만 배치하되, 기술 스택이나 요약에 명시된 것들만 포함시키는 것이다.

ROS2와 Gazebo가 빈약한 건 조교님의 기대와 다를 수 있지만, 실제 데이터를 보니 EtherCAT/CAN, 전력전자/모터드라이브, 하드웨어 분야는 풍부하고, ROS2는 농업 로봇 정도 3개, Gazebo는 1개 정도만 있다.

Description을 더 자세히 확인해야 할 것 같다. 특히 AI자율주행, 로보컵, 안내견 프로젝트 같은 것들이 시뮬레이션을 사용하는지 명확하지 않으니까, 전체 description을 살펴봐서 정확한 분류를 해야겠다. 이제 grep으로 description 전체를 검색해서 EtherCAT, CAN, ROS2, ROS, Gazebo, Isaac, 강화학습, RL, Sim2Real 같은 키워드들을 매칭시켜보자.데이터 다 확인했다. 분류 정확도를 위해 6개 카테고리 키워드를 description까지 포함해 전수 검색한다.
```
This block is not supported on your current device yet.
```

키워드 매칭 결과를 검토하니 몇 가지 거짓 양성이 보인다. "시뮬레이션"이 카테고리3에서 여러 항목을 잡았지만 실제로는 Gazebo나 Isaac과 무관할 수 있고, 카테고리1의 "spi"는 복싱 관련 항목을 포함했는데 이건 실제 SPI 통신과는 다르다.

정확한 분류를 위해 모호한 항목들의 설명을 직접 확인해야 한다. 특히 카테고리3의 [9], [11], [20], [26]에서 "시뮬레이션"이 정확히 무엇을 의미하는지, 카테고리2의 [5]가 ROS2인지 ROS인지, 그리고 카테고리1에서 SPI/RS485를 포함할지 여부를 판단해야 한다. 조교님이 제시한 카테고리명이 "전기 인터페이스 / EtherCAT·CAN 통신"이므로 EtherCAT/CAN이 핵심이지만 통신 인터페이스도 포함되는 것 같다.

각 프로젝트의 설명을 살펴봐야 정확한 분류가 가능할 것 같다. [18], [13], [24]가 실제로 EtherCAT, CAN, 주요 통신 프로토콜에 해당하는지 description을 확인해야겠다.모호한 매칭(특히 "시뮬레이션", SPI/RS485)을 검증하려고 해당 프로젝트 description을 확인한다.
```
This block is not supported on your current device yet.
```

이제 정확한 분류가 가능해졌다. 검증 결과를 보니 카테고리 3의 거짓 양성들을 정리할 수 있다. 농기구 추종 항목은 실제로 Gazebo 시뮬레이션을 사용하고 있지만, 공유모빌리티 항목은 MATLAB Simulink을 사용하고 있어서 다시 분류해야 한다.

플라즈마, 30kV, AI자율주행 항목들도 검토했는데, 플라즈마와 30kV는 각각 ANSYS와 EMI 시뮬레이션으로 Gazebo/Isaac과 무관하고, AI자율주행은 시뮬레이션에서 실차로 가는 sim-to-real 접근이지만 구체적인 도구가 아직 정해지지 않았다. 결국 명시적으로 Gazebo/Isaac Sim을 사용하는 프로젝트는 농기구 추종 하나뿐이고, AI자율주행은 진행 중인 상태다.

이 현실을 정직하게 반영해야 하는데, 카테고리 3이 상당히 빈약해 보인다. ROS2 카테고리를 다시 살펴보니 농기구 추종과 시설농가 AMR은 명확하게 ROS2를 사용하고 있고, 안내견 프로젝트는 RPLiDAR와 YDLidar를 ROS 노드로 사용하면서 /scan과 RViz로 검증하고 있는데, ROS2인지 명시되지 않았다.

로보컵과 안내견 프로젝트의 강화학습 부분을 보니, 로보컵은 강화학습을 탐색 방향으로만 언급했고 실제로는 ZMP와 MPC가 주요 기술이며, 안내견은 ARS(Augmented Random Search)로 학습하면서 MuJoCo를 사용하고 있다.

로커보기는 14차원 파라미터 최적화를 하는데 이건 강화학습이 아니라 블랙박스 최적화 문제고, JAX와 CUDA 기반 시뮬레이션 최적화를 사용하고 있다. 따라서 강화학습/Sim2Real 카테고리에는 안내견 프로젝트가 명확하게 해당되고, 로보컵은 약하게만 포함된다.

이제 통신 인터페이스 카테고리를 정리하면, EtherCAT는 버섯SCARA에서 사용했고, CAN 통신은 여러 프로젝트에서 활용했는데 MPPT 보드는 상용 제품이라 제외하고, RS485는 농기구에서 STM32로 구현했으며, SPI는 복싱 프로젝트에서 보드 내부 통신으로 사용했다. 고전압 인버터, MPPT, 회생제동, BLDC 모터드라이버, EV 파워트레인 등이 확실한 항목들이고, ODrive 모터제어나 SMPS 전원회로 같은 것들은 약하게 포함시키되 모터드라이브 자체가 아닌 것들은 구분해야 한다.

카테고리 6에서는 실제 하드웨어 제작과 실증이 핵심인데, 드론 자작부터 EV 실차 개조 성공, 벽면등반 로봇, SCARA 로봇 같은 것들이 대표적이다.

분류 전략을 다시 정리하면, 조교가 제시한 6개 카테고리의 데이터 분포가 불균등해서 EtherCAT/CAN은 탄탄하지만 Gazebo/Isaac 시뮬레이션 쪽은 매우 빈약한 상황이다.

전력전자와 모터드라이브가 10개 이상으로 가장 많고, 결국 거의 모든 프로젝트가 실제 하드웨어 제작과 실증을 기반으로 하고 있다는 점이 핵심이다. 조교의 요청대로 정직하게 이 분포를 보여주되, 빈약한 카테고리를 억지로 부풀리지 않는 게 맞다.

결과물은 HTML 문서로 6개 섹션을 만들고, 각 섹션마다 카테고리 제목과 한 줄 설명을 붙인 후 해당 프로젝트 카드들을 배치한다. 카드에는 썸네일, 제목, 연도, 요약, 기술 태그, 그리고 깃허브나 유튜브 링크를 포함시킨다. 한 프로젝트가 여러 카테고리에 나타날 수 있으니 각 맥락에 맞춰 설명을 다르게 작성하면 된다.

이미지 URL은 showcase.zetin.u-os.ac.kr/ 기본 경로에 상대경로를 붙이는 방식으로 구성하고, onerror 폴백을 설정해두면 이미지가 로드되지 않아도 레이아웃이 유지되고 사용자가 브라우저에서 실제 상황을 확인할 수 있다. 검증을 위해 web_fetch를 시도할 수도 있지만, 이미지 바이너리 처리의 효율성을 고려하면 onerror 처리로 충분하다. 안내 메시지에 이 부분을 명시해두겠다.

이제 KILL.md를 확인한 후 HTML을 작성해야 한다. 조교 요청이 기술 카테고리별 정리와 사진 포함이므로 HTML 아티팩트가 최적이다. 마크다운은 원격 이미지 렌더링이 환경마다 다르고, docx는 도메인 차단으로 이미지를 못 넣기 때문이다. frontend-design을 참고해서 진행하겠다.분류 검증 끝났다. 데이터 분포가 카테고리별로 크게 다르다 (전력전자·하드웨어는 풍부, Gazebo/Isaac은 빈약). 이걸 그대로 반영해서 HTML 정리 페이지를 만든다. UI 작성 전 디자인 스킬부터 본다.
```
This block is not supported on your current device yet.
```

프론트엔드 디자인 가이드를 검토했고, 이제 ZETIN 로봇 동아리의 프로젝트들을 정리한 포트폴리오 자료를 만들어야 한다는 걸 파악했다. 하드웨어, 전력전자, 임베디드, 로봇 기술 중심으로 조교와 연구실에 보여줄 수 있는 형태로 카테고리별로 구성해야 하니, 먼저 전체 디자인 방향을 결정해야 한다.

"ZEro To INfinite"라는 동아리명의 의미와 청중(기술자들)을 고려하면, 엔지니어링 문서 같은 톤으로 가되 너무 뻔하지 않아야 한다. 28개 프로젝트를 6개 기술 카테고리로 나누고 각각 사진과 함께 표시하는 게 핵심이다. AI 디폴트 색상 조합들을 피하면서도 회로기판이나 전력전자 느낌을 살려야 하는데, 다크 테크니컬은 흑배경+네온이 너무 흔하니 엔지니어링 노트 감성으로 가는 게 낫겠다. 옅은 그레이나 오프화이트 배경에 ZETIN의 정체성을 담은 강한 액센트 색 하나를 선택하는 방향으로 생각 중이다.

폰트는 기술 문서의 위계를 살려서, 디스플레이용 기하학적 산세리프와 본문용 깔끔한 산세리프, 그리고 스펙이나 데이터용 모노스페이스를 조합하되, 한글이 많으니 Pretendard 같은 한영 혼용 폰트를 고려하고 있다.

HTML 아티팩트는 브라우저에서 렌더되므로 Google Fonts나 Pretendard의 CDN 링크를 직접 사용할 수 있고, 보통 문제없이 작동한다. Pretendard를 메인으로 쓰되 영문 라벨이나 숫자에 모노 폰트를 추가하는 방식으로 가면 충분할 것 같다.

이제 레이아웃을 구성하는데, 헤더에 프로젝트 제목과 부제, 그리고 전체 구조를 배치하는 부분부터 시작하려고 한다.

전체 28개 프로젝트를 6개 카테고리로 나누고, 각 섹션마다 카테고리 번호와 제목, 설명을 배치한 뒤 그 아래에 프로젝트 카드를 그리드로 배열한다. 카드는 썸네일, 제목, 연도/소속, 요약, 기술 칩, 링크로 구성하고 반응형으로 만들어서 데스크탑에선 2-3열, 모바일에선 1열로 표시되게 한다. 카테고리 번호를 크게 표시해서 시각적 시그니처로 활용할 수 있을 것 같은데, 정확한 디자인 방향을 다시 확인해야 할 것 같다. 카테고리별 프로젝트 수를 헤더에 작은 막대 차트나 카운트로 표시해서 강점이 어디에 집중되어 있는지 한눈에 보이게 하고, 각 섹션 헤더에도 그 수를 크게 표시하는 방식으로 가자. 색상은 전기/전자 모티프를 살려서 전기 블루나 호박색 팔레트를 쓰는 게 좋을 것 같다.

다크 네이비 베이스에 시안/블루 액센트를 쓰거나, 아니면 라이트 방향으로 오프화이트 배경에 잉크 네이비 텍스트와 일렉트릭 블루나 호박색 액센트를 조합하는 방식도 있다. 호박색이 전류와 열, 에너지 느낌을 더 잘 살려내니까 회로 녹색보다 낫겠다.

지금 팔레트를 구체적으로 시도 중인데, 중성 페이퍼 톤의 배경에 거의 검정에 가까운 잉크, 그리고 액센트로 전기 주황을 고려하고 있다. 다만 산성 그린이나 버밀리언, 테라코타 같은 색들은 디폴트 경고 느낌이 나니까 피해야 하고, 대신 전기 인디고나 바이올렛 같은 대안도 살펴보는 중이다.

울트라마린이나 일렉트릭 블루 계열이 공학적이고 진지한 느낌을 줄 수 있을 것 같아서, 차가운 오프화이트 배경에 흰색 카드와 얇은 보더를 조합하면서 액센트 색을 최종 결정하려고 한다.

타이포그래피는 한글과 영문 본문에 Pretendard를 쓰되, 데이터나 스펙 같은 기술적인 부분은 모노스페이스 폰트로 처리해서 칩이나 부품번호 같은 요소들에 기술감을 더하고, 디스플레이 텍스트는 Pretendard의 무거운 웨이트로 강조할 계획이다.

이제 레이아웃 구조를 설계 중인데, 헤더에 로고와 제목을 배치하고 우측에 메타 정보를 넣은 뒤, 카테고리별 분포를 시각적으로 보여주는 막대 그래프를 시그니처 요소로 활용하고, 그 아래로 각 섹션을 번호와 함께 정렬해서 프로젝트들을 카테고리별로 구분할 예정이다. 프로젝트들 중 이미지가 없는 것들은 텍스트나 패턴으로 썸네일을 대체하고, 각 카테고리별 프로젝트 배치를 확정하되 중복을 허용하면서 카테고리 헤더에서 맥락을 제공하고 프로젝트 설명은 공통으로 사용하는 방식으로 진행하려고 한다.

이제 전기 인터페이스 카테고리부터 시작해서 EtherCAT, CAN, RS485 통신을 다루는 프로젝트들을 정리하고 있다.

ROS2 제어 통합 부분에서는 자율주행과 경로 추종 관련 3개 프로젝트를 보고 있는데, 농기구 추종과 시설농가 AMR, 안내견 4족 로봇이 포함되어 있다. Gazebo와 Isaac Sim 시뮬레이션은 현재 1개 프로젝트가 진행 중이고, AI 자율주행 프로젝트는 2026년에 진행될 예정이다.

강화학습과 Sim-to-Real 영역에서는 2개 프로젝트가 있는데, 안내견 4족은 MuJoCo 시뮬레이션에서 학습한 정책을 실제 로봇에 적용하는 방식을 취하고 있고, 로보컵 2족은 보행 궤적 생성에서 강화학습을 탐색 방향으로 활용하면서 ZMP가 주요 제어 방식이다. 이제 전력전자와 모터드라이브 영역으로 넘어가는데 여기서는 11개 프로젝트가 있다.

모터드라이브 시스템부터 시작해서 화물 보조, 공유모빌리티의 3상 인버터와 BLDC, 채소 절단기의 BLDC 드라이버, 매트릭스 배터리, ANN 기반 MPPT의 Buck 컨버터, 30kV DC/DC의 Push-Pull과 CW 토폴로지, 플라즈마 큐어링의 고전압 인버터, 리프트 하강의 회생제동과 게이트드라이버, EV 컨버전의 인버터/DC-DC/BMS, 그리고 로커보기의 ODrive BLDC 드라이버까지 다양한 전력전자 응용이 포함되어 있다. 이 영역이 프로젝트 포트폴리오의 최대 강점이다.

인트레이서도 속도제어 관련이지만 상대적으로 약해 보이니 제외하거나 포함할지 고민 중이다. 현재 11개 프로젝트로 충분하고, 인트레이서는 06번 섹션으로 분류하는 게 나을 것 같다.

하드웨어 제작 및 실증 섹션을 정리하면서 거의 모든 프로젝트가 실제 PCB와 물리적 구현을 포함하고 있다. EV 컨버전은 실차 개조로 50km/h 주행에 성공했고, 벽면 등반 로봇은 비파괴검사용 실물, 극한환경 로커보기는 험지 주행 플랫폼, 버섯 SCARA는 5축 실물, ZETIN 드론은 자작 PCB와 비행 성공, 디지털 트윈 복싱은 Altium PCB 설계 등이 대표 실증 사례다.

목록이 너무 길어지고 있으니 06 섹션은 "전 프로젝트가 직접 제작·실증 기반"이라는 메타 메시지를 강조하면서 대표 사례 6~8개만 엄선해야 한다. 다른 카테고리와의 중복을 피하고 실차, 실물, 필드 검증이 핵심인 프로젝트들을 우선적으로 선별하는 방향으로 진행 중이다. 이제 각 프로젝트 카드에 들어갈 핵심 정보를 정리하고 있다. 공통 데이터 구조는 제목, 연도, 소속, 한 줄 요약, 핵심 기술 3-4개, 링크, 썸네일로 통일하고, 대표 사례 8개를 선정해서 카드용 요약을 압축 중이다. 드론은 자체 펌웨어와 무선제어 기능을, 로커보기는 험지 주행 능력을 핵심으로 정리하고 있다. [7] 로봇팔 관절 액추에이터를 직접 제작하면서 전류 기반 토크 제어와 충돌 감지 기능을 구현하고, [8] 딥러닝 비전으로 버섯을 판별해 5축 SCARA 로봇이 수확하도록 했으며, [9] 킥보드와 자전거의 배터리·모터 시스템을 역설계해 공유 모빌리티에 적용했다. [11] SDBD 오존 플라즈마 발생기를 8kV 고전압 인버터로 구동하고 AI로 고장을 진단하며, [12] 32개 셀의 배터리를 실시간으로 직렬·병렬 전환해 200V 전력을 관리하는 시스템도 개발했다.

[13] NPU와 MCU의 인공신경망으로 부분 음영 상황에서 최대전력점을 추종하는 초저전력 MPPT 보드를 만들고, [14] 파종기의 기어 회전을 근접센서로 감지해 결주를 간접 검출하는 저비용 STM32 기반 솔루션을 구현했다. [15] 배달로봇의 리프트에 회생제동 과전압 클램핑 기능을 가진 부착형 보드를 적용하고, [16] 볼스크류 엣지 상태를 AI로 실시간 진단하는 자체 PCB를 PoE 지원으로 설계했으며, [17] LSTM 오토인코더로 진동과 온도 데이터를 분석해 고장을 예측하는 또 다른 PCB도 개발했다.

가속도 센서 데이터를 Unity 디지털 트윈으로 시각화하면서 자세를 점수화하고, [19] 기타 타브 채보를 위해 현과 프렛의 접촉을 전기적으로 감지해 운지를 특정하는 마이크 없는 시스템을 만들었다. [20] Push-Pull 토폴로지와 21단 배전압 회로로 30kV DC/DC 컨버터를 자체 설계했고, [21] 무한궤도와 다빈치 캠 메커니즘으로 채소 절단을 자동화했으며, [22] 마티즈 엔진을 제거하고 20kW PMSM과 144V 배터리팩을 장착해 50km/h 주행이 가능한 EV로 개조했다. [23] 흡입모터 3개를 회전시켜 벽면에 흡착하는 로봇으로 벽면 이동과 비파괴 결함검사를 수행하고 있다.

이제 [25] ATmega128 기반 전류제어로 토크를 보조하는 전동 화물캐리어 드라이브를 구현하고, [26] 학습 기반 인지·계획·제어 파이프라인으로 자율주행을 진행 중이며, [27] 신선도 유지 쿨링팬의 자가진단과 예지보전 시스템도 2026년까지 개발할 예정이다. 각 프로젝트에서 핵심 기술 3~4개를 선별해 카드 칩용으로 정리한 후 HTML로 작성해 데이터를 JavaScript로 연결할 준비를 하고 있다.

28개 프로젝트를 6개 카테고리로 매핑하는 구조를 정의해서 프로젝트 객체와 카테고리별 ID 리스트를 만들어 동적으로 렌더링할 계획이다. 아티팩트 HTML은 단일 파일이고 브라우저에서 JavaScript가 정상 실행되므로 동적 생성이 문제없다. 이미지 경로는 showcase.zetin.uos.ac 도메인에서 가져올 예정이다.

이미지 로드 실패 시 그라데이션 배경과 제목 이니셜 또는 플레이스홀더를 표시하는 폴백을 구현하고, Python 스크립트로 projects.json의 정확한 데이터(제목, 연도, 소속, 요약, 기술, 썸네일, 링크)를 읽어서 카테고리 매핑과 함께 HTML에 임베드하는 방식이 가장 안전하다.

카드 설명으로는 summary를 그대로 사용하고(대부분 1문장이라 적당함), 기술 스택은 앞 4개만 표시하는 것으로 결정했다.

약한 매칭이 필요한 항목들(ROS, RL탐색, RS485, 진행중 프로젝트)은 카테고리 헤더나 카드에 작은 노트로 표기할 생각인데, 정직성을 어떻게 표현할지 고민 중이다. 이제 Python 스크립트로 HTML을 생성할 건데, projects.json을 로드해서 카테고리별로 정리하고 카드 형태로 렌더링하겠다. 헤더에는 전체 프로젝트 28개, 연도 범위, 그리고 6개 카테고리의 분포를 수평 막대로 표시할 예정이고, 스타일은 Pretendard 폰트에 일렉트릭 블루 액센트와 라이트 페이퍼 배경으로 맞추겠다.

카테고리 카운트는 각 카테고리에 배치된 프로젝트 수를 세는데, 프로젝트가 여러 카테고리에 들어갈 수 있으니 중복을 포함해서 센다. [10] 펌프 고장진단과 [27] 쿨링팬 같은 경우는 명확한 카테고리 매칭이 애매한데, 펌프는 AI 이상탐지와 센서보드 조합이라 06 하드웨어 실증에 넣을 수 있고, 쿨링팬도 진행 중인 프로젝트라 06에 배치하는 게 맞을 것 같다.
