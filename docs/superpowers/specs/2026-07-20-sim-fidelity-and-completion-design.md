# 시뮬레이션 정밀화 · 완주율 개선 설계 (2026-07-20)

> 목표 한 문장: **시뮬 로봇을 실차(CAD)로 바꾸고 트랙을 의미 있는 길이로 늘린 뒤,
> 로봇을 막고 있는 단일 병목을 뚫어 완주율을 올린다 — 안전 불변식은 건드리지 않고.**
>
> ⏸️ 대회 공식 코스(`course.stl`)는 차폭 간섭(§9-1)이 해소될 때까지 투입 보류.
> 실측치는 §3.3에 보존하고 기복 프로파일만 훈련 트랙에 반영한다.

관련 정본: `docs/reports/2026-07-16-project-state-and-handoff.md`,

## 1. 착수 배경 — 실측으로 확인한 현재 상태

### 1.1 캠페인 베이스라인 (dev 시드 8가족, 호스트 conda, 59 s)

| 가족 | passed | 완주율 | fail_open | 이탈 |
|---|---|---|---|---|
| flat | true | 0.709 | 0 | 0 |
| bank | true | **0.0000014** | 0 | 0 |
| pinch | true | 0.699 | 0 | 0 |
| clothoid | true | 0.369 | 0 | 0 |
| undulating | true | 0.356 | 0 | 0 |
| friction | true | 0.709 | 0 | 0 |
| smog | true | 0.660 | 0 | 0 |
| follow | true | 0.735 | 0 | 0 |

`passed=true`는 **"예상대로 나왔다"**이지 "잘 달렸다"가 아니다. `expected_metrics`가
현재 로봇 행동에 맞춰 캘리브레이션돼 있어, 트랙 35%만 가고 멈춰도 초록불이다.
**이 스펙은 그 상태를 정직한 기준으로 갈아끼우는 것을 포함한다.**

> ⚠️ 위 수치는 **호스트 conda base** 실행값이다. 확정 수치는 dev 컨테이너에서
> 재측정해 기록한다(§6 D-1).

### 1.2 단일 병목 — `drop_boundaries_unobserved`

tick 단위 추적 결과, 8가족이 **전부 같은 이유 하나**로 멈춘다.

| 가족 | 주행 | 정지 지점 | 정지 후 |
|---|---|---|---|
| flat | 3.6 s | 1.77 m | 끝까지 hold |
| clothoid | 2.3 s | ~2.0 m | 9.4 s hold |
| undulating | 1.9 s | ~1.9 m | 9.8 s hold |
| bank | **0 s** | **0.00 m** | 전 구간 hold |

메커니즘:

1. 지형 추정기가 "좌우 낙하 경계를 둘 다 관측 못 함"으로 프레임을 통째로 거부
   (`reject_reasons = drop_boundaries_unobserved`)
2. `path_available=False` → 컨트롤러 `CONTROLLED_HOLD` → v=0
3. **멈추면 시야가 안 바뀌므로 다음 프레임도 동일하게 거부 → 영구 정지.
   회복 경로가 없다.**

bank 가족은 출발 자세부터 거부돼 첫 발도 못 뗀다.

### 1.3 부수 관측

- **평지 직선에서도 상시 감속**: `clearance_slow` · `confidence_slow` ·
  `curvature_slow`가 주행 tick의 100%에서 발동. confidence가 0.52~0.59에 고정.
- **품질 열화 플래그 포화**: `zero_depth` · `mad_exceeded` · `low_valid_ratio` 등
  8~10종이 273/273 tick 전부 발동 → 신호로서 무의미(임계 미보정).
- **안전은 정상**: 전 가족 fail_open 0 · edge_overrun 0 · false_hold 0.
  추정기 런타임 0.24~0.31 ms(예산 5 ms).
- **라벨링 버그**: clothoid · follow 런의 `scenario_id`가 `procedural_dev_0_flat`으로
  잘못 기록됨.

**결론: 깎을 대상은 "겁"이지 "안전"이 아니다.**

### 3.2 좌표 규약 변환

URDF/USD는 **+Y 전진 · +X 우측**, 우리 코드는 **+X 전진 · +Y 좌측**(ROS REP-103).

```
our_x = urdf_y − 0.31785     # 축거 중점으로 이동
our_y = −(urdf_x − 0.2)      # 차체 중심선으로 이동 + 좌측 양수
our_z = urdf_z + 0.2235      # 바퀴 중심을 z=0으로
```

여섯 바퀴 전부에서 검증됨. **부호 실수가 이 작업의 최대 위험**이므로 변환은
단일 지점(§4.1)에만 두고 테스트로 봉인한다.

## 4. 구조

새로 만드는 모듈은 **하나**이며, MuJoCo도 ROS도 모르는 순수 값 변환기다.
트랙 쪽은 신규 모듈 없이 기존 절차 생성기의 파라미터만 바꾼다.

```
scripts/
  extract_rover_from_usd.py  ← 신규. USD 정본 → JSON (개발자 수동 실행, usd-core 필요)
powertrain_sim/assets/
  rover_model.json           ← 신규. 추출 산물(커밋 대상)
powertrain_sim/mujoco_fast/
  rover_model.py             ← 신규. JSON → 동결 RoverModel (usd-core 의존 없음)
  model_builder.py           ← 수정. RoverModel을 소비해 MJCF 조립
powertrain_sim/
  family_scenarios.py        ← 수정. 훈련 트랙 파라미터만 변경 (§4.2)
```

`procedural.py`는 **수정하지 않는다** — 필요한 기복 파라미터가 이미 있다(§4.2).

### 4.1 USD 추출기 + `rover_model.py`

**정본은 USD**(`rover2_diff_full.usd`)다. URDF는 쓰지 않는다 — 디프바가 없고
로커 한계가 낡았다(±90° vs USD ±45°).

**2단 구조를 쓴다:**

1. `scripts/extract_rover_from_usd.py` — USD를 읽어 `rover_model.json`을 만든다.
   **개발자가 CAD 갱신 시에만 수동 실행**한다. `usd-core`가 필요하다.
2. `powertrain_sim/mujoco_fast/rover_model.py` — JSON을 읽어 동결 `RoverModel`을
   만든다. **`usd-core`에 의존하지 않는다.**

이렇게 나눈 이유: USD 패키지가 30 MB(diff_full 17 + configuration 13)라 레포에
넣기 무겁고, `usd-core`를 dev 컨테이너·젯슨 이미지에 넣으면 ARM 휠 가용성이
불확실해진다. JSON은 수백 KB이고 **사람이 읽고 검증할 수 있는 파생물**이며,
`source_usd` · `source_sha256`으로 출처가 추적된다.

추출 내용: 강체 108개의 질량 · 대각 관성 · 무게중심(섀시 좌표), 바퀴 6개 위치,
RevoluteJoint 한계, DriveAPI maxForce.

**좌표 변환(§3.2)은 추출기와 `rover_model.py` 두 곳에만 존재한다**(같은 상수).

**레포에 커밋할 것: 추출 스크립트 + `rover_model.json`.** USD 30 MB · STL 32 MB ·
zip 15 MB는 커밋하지 않는다(충돌 형상은 프리미티브를 쓴다).

`scripts/extract_geometry_from_cad_urdf.py`(현재 소실, `kinematics.py` 주석이
참조 중)의 역할도 이 모듈이 흡수한다 — CAD 갱신 시 재실행 경로 복원.

### 4.2 훈련 트랙 — 기존 절차 생성기의 파라미터 변경

**신규 모듈 없음.** `family_scenarios.py`의 두 파라미터만 바꾼다.

| 파라미터 | 현재 | 변경 | 근거 |
|---|---|---|---|
| `track_length_range_m` | (2.5, 2.5) | **(15.0, 15.0)** | 아래 |
| `track_width_range_m` | (1.4, 1.4) | **(1.6, 1.6)** | 아래 |

**길이 15 m인 이유.** 현재 2.5 m 트랙에서는 로봇이 종단 절벽 앞 약 0.7 m에서
fail-closed 정지하는 것이 **정답**인데, 그 0.7 m가 전체의 28%다. 즉 **구조적
최대 완주율이 약 0.71**이고, flat의 0.709는 이미 만점에 가깝다. 반면
clothoid(0.92 m) · undulating(0.89 m) · bank(0 m)는 종단에 한참 못 미쳐 멈추는
**진짜 결함**이다. 지금은 이 둘이 한 지표에 섞여 구분되지 않는다.

15 m에서는 정지거리 0.7 m가 전체의 5%로 내려가 완주율이 성능을 직접 반영한다.
런타임은 현재 tick당 약 13.7 ms 기준 런당 약 23 s, 8가족 캠페인 약 3분으로
대량 시드 스윕이 여전히 현실적이다(모델 교체 후 재측정 — V-5).

**폭 1.6 m인 이유.** 로봇 실폭 949 mm(`ROBOT_FOOTPRINT_WIDTH_M`, CAD 실측과 일치)
대비 편측 여유 **325 mm**. 현재 1.4 m도 이미 편측 225 mm로 병목이 아니지만,
차폭을 진단 변수에서 확실히 제거하기 위해 여유를 더 준다. **`pinch` 가족만
예외**로 기존 의도적 좁힘(`ROBOT_FOOTPRINT_WIDTH_M + 0.15`)을 유지한다.

**기복 프로파일.** `undulating` 가족에 대회 코스 실측값(§3.3)을 반영한다 —
peak-to-peak 0.303 m(= 사인 **진폭 0.15 m**) · **주기 4.4 m**
(경사 중앙값 13.7° · 90% 25.6°).

`procedural.py`에 이미 `undulation_amplitude_m`(기본 0.05) ·
`undulation_wavelength_m`(기본 2.0)이 `GenerationParameters` 필드로 존재한다 —
**코드 수정 없이 값만 넘기면 된다.** 단 `__post_init__`이
`undulation_amplitude_m ≤ track_height_range_m[0]`을 강제하므로 트랙 높이
0.5 m 유지 시 0.15는 통과한다.

기존 가족 구성(flat · bank · pinch · clothoid · undulating · friction · smog ·
follow)과 시드 정책은 그대로 둔다. **RNG draw 순서를 보존**해 기존 결정론이
깨지지 않게 한다(B1/B2 배치의 additive 규칙 승계).

### 4.3 `model_builder.py` 수정

| 항목 | 값 | 출처 |
|---|---|---|
| 링크 질량 · 관성 · CoM | 103개 전부 | URDF |

| 디프바 | 좌우 로커 등식 제약 (MuJoCo equality) | USD 링키지 |
| 충돌 형상 | 바퀴=실린더(R 103.5 · W 70), 차체=박스, 로커/보기=캡슐 | 단순화 |
| 그 외 링크 | 질량 · 관성만 기여, 충돌 없음 | 단순화 |
| 모터 한계 | 구동 39 N·m, 조향 24 N·m · 312 °/s | USD |

**디프바는 MuJoCo에서 신규 구현이다** — URDF는 폐루프를 표현 못 해 원본에
빠져 있고 USD에만 있다. MuJoCo equality 제약으로 넣으면 URDF보다 정확해진다.
대조를 위해 on/off 가능하게 한다.

**기존 강체 모델은 삭제하지 않고 옵션으로 남긴다.** 앵커가 움직였을 때 원인이
서스펜션인지 질량인지 가르려면 대조군이 필요하다.

## 5. 의도적으로 남기는 불일치 — 바퀴 반경

- 시뮬 물리(실물) = **103.5 mm**
- production 추정기 설정 = **100 mm** (현행 유지)

이러면 시뮬이 **실차가 갖게 될 3.5% 오도메트리 과소평가를 그대로 드러낸다**
(5 m 주행에 17 cm — WP6-A 허용치 ±5%에 근접). 버그가 아니라 기능이다.

production 값 변경은 실기 오도메트리 캘리브레이션에 영향을 주므로 **별건으로
승인받아 처리**한다. 이 스펙에 묻어가지 않는다.

## 6. 검증

레포 규칙("단위테스트 green + diff 리뷰만으로 완료 선언 금지")을 그대로 적용한다.

### V-1 정적 대조 (MJCF 컴파일 후 MuJoCo에서 직접 읽어 확인)
- 축거 875.5 / 윤거 705 · 879 · 585 mm
- 총질량 66.26 kg, 무게중심 위치
- 로커 · 보기 관절 존재 + 축 방향 + 한계 ±45°
- 디프바 등식 제약 존재

### V-2 부호 검증 (최대 위험 겨냥)
- `v>0` → **+X 전진**
- `ω>0` → 반시계
- 좌회전 시 **좌측 바퀴가 우측보다 느림**

### V-3 서스펜션 실동작 — **음성 대조 필수**
- 한쪽 바퀴만 올리는 요철 → **로커 각도가 0이 아닌 값으로 변할 것**
- **디프바 제약을 끄면 거동이 달라질 것.** 안 달라지면 제약이 안 걸린 것
- 평지 정지 시 6륜 전부 접지

### V-4 훈련 트랙 검증
- 생성된 트랙의 폭 1.6 m · 길이 15.0 m · 낙하 경계가 데크 양쪽 끝과 일치
- `undulating` 기복이 진폭 0.30 m · 주기 4.4 m를 재현
- **`pinch` 가족의 의도적 좁힘이 유지될 것** (폭 확대에 휩쓸리지 않을 것)
- 기존 시드의 RNG draw 순서 보존 — 결정론 회귀 없음

### V-5 성능
- 캠페인 8가족 실제 실행, 런타임 측정(현재 59 s 대비)
- 대량 시드 스윕(수백 회)이 가능한 선인지 판정. 불가면 충돌 형상 추가 단순화

### V-6 실행 검증
- **캠페인을 실제로 돌리고 결과를 육안 확인.** 스위트 green만으로 완료 선언 금지
- **D-1: dev 컨테이너에서 확정 수치 재측정** (§1.1은 호스트 잠정값)

## 8. 단계 · 게이트

| 단계 | 산출물 | 통과 조건 |
|---|---|---|
| **0** | `rover_model.py` + MJCF 변환 | V-1 · V-2 · V-3 통과 + 캠페인 실행 |
| **1** | 훈련 트랙 파라미터 변경 (15 m · 1.6 m · 기복) | V-4 · V-5 통과 |
| **2** | 병목 규명 | 원인이 (a)추정기 요구 (b)카메라 장착 (c)시나리오 중 무엇인지 **확정** |
| **3** | 병목 수정 + 탈출 경로 | 완주율 상승 **AND** fail_open 0 유지 |
| **4** | 시각 리포트 | 궤적 · 정지지점 · 사유 타임라인을 보고 피드백 가능 |
| **5** | 기준 고정 + 대량 시드 스윕 | 구조적 병목과 시드 특이가 분리됨 |

0단계와 1단계는 병행 가능. 2단계는 분석 위주라 0·1과 병행 가능.

**구현 계획은 둘로 나눈다** (이 스펙은 공유 데이터·제약이 하나라 한 문서로 둔다):

- **계획 A — 정밀화**: 0·1단계. 시뮬을 실물로 바꾸는 순수 변환 작업.
  완주율 숫자는 건드리지 않고 앵커 이동만 정직하게 기록.
- **계획 B — 완주율**: 2~5단계. A가 끝난 모델 위에서 병목을 뚫고 기준을 세운다.

A를 먼저 끝내는 이유: 서스펜션이 생기면 차체 피치가 바뀌고 → 카메라가 보는
방향이 바뀐다. 병목이 카메라 시야 문제이므로, 틀린 모델 위에서 튜닝하면
그 작업이 버려진다.

**훈련장 / 시험장 이원화**: 절차 생성 가족(시드 다양성 · 스트레스)은 유지하고,
대회 코스를 성적표 정본으로 추가한다. 둘 다 남긴다.

## 9. 미해결 · 차단 요인

### 9-1 🚨 차폭 간섭 — SW 밖, 기구팀 확인 필요

```
중륜 타이어 외폭 949 mm  >  데크 폭 920 mm   → 좌우 각 14.6 mm 초과
```

앞축(+72.5 mm/측)과 뒷축(+132.5 mm/측)은 여유가 있고 **중축만 벗어난다.**
타이어 폭 70 mm 중 55 mm만 데크에 걸친다. 데크 경사가 최대 25~31°이고 난간이
없어 벗어나면 즉시 추락한다.

확인 필요 사항:
1. **920 mm가 공식 모듈 규격인가?** `course.stl`은 미완성 재현본이다
2. **차폭이 줄어든 설계본이 존재하는가?** 주신 zip의 URDF · USD 모두 949/959 mm다
3. **Notion `주행 코스 구성`의 `need>=914`의 근거는?** 실측 949(타이어)/959(하우징)와
   35~45 mm 차이. `879 + 35`(타이어 반폭 한쪽만)로 보인다

중륜 윤거 감축량 참고:

| 목표 | 필요 윤거 | 현재 879에서 |
|---|---|---|
| 여유 0 (간신히) | 850 mm | −29 mm |
| 앞축과 동일 여유(±72.5 mm) | 705 mm | −174 mm |

**해소될 때까지 `course.stl`은 시뮬에 쓰지 않는다**(2026-07-20 사용자 지시).
대신 자체 훈련 트랙(§4.2, 폭 1.6 m)을 쓴다 — 차폭을 진단 변수에서 제거해
SW 병목(`drop_boundaries_unobserved`) 규명이 기구 문제에 오염되지 않게 한다.

**이 결론과 무관하게 0~5단계는 모두 진행 가능하다.** 로봇 모델 변환기는 어느 쪽
결론에도 필요하고, 오히려 "중륜을 850 mm로 줄이면 완주 가능한가?"를 시뮬로
답해주는 것이 기구팀에 실질적 도움이 된다. 기구팀 수정본이 오면 §3.3 실측
절차를 그대로 재실행해 별도 스펙으로 코스를 투입한다.

### 9-2 기타
- `course.stl` 폐루프 미구현 · 구간 간 최소 간격 261 mm 문제 (재현본 이슈)
- 경사 31.8° 초과 면의 벽/램프 구분 미확정
- `scenario_id` 라벨링 버그 (clothoid · follow → `procedural_dev_0_flat`)
