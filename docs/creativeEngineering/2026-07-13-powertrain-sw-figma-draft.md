## 5. 통신 · 관제

### 5-A. CAN 단일 채널 통합 제어 [완료 · 실기 검증]

**개념** — 조향 4축(CubeMars AK45-36)과 구동 6축(BL70200 + ODrive 듀얼축 보드 3장), **총 10개 모터를
단 하나의 CAN 버스(`can0`, 500 kbps)** 에 데이지 체인으로 물려 젯슨 한 대가 직접 제어한다. 서로 다른
프로토콜(AK = VESC 확장 프레임 / ODrive = CANSimple 표준 프레임)이 같은 물리 버스를 공유하며,
ID 대역(조향 1~4 / 구동 11~16)으로 충돌 없이 공존한다.

**노이즈 내성 설계** — ① **트위스티드 페어 + 차동 전송**: CAN은 CAN_H/CAN_L의 전압 차로 비트를 읽으므로,
꼬임쌍선에 실린 공통모드 노이즈(모터 PWM 스위칭 노이즈)는 두 선에 똑같이 실려 차분에서 상쇄된다.
② **절연형 트랜시버(ADM3053)**: 젯슨 접지와 모터 파워 접지를 갈바닉 절연해 그라운드 도메인을 분리한다.
③ **양단 60 Ω 종단 + 500 kbps**: 긴 케이블 길이를 줄일 수 없으므로 비트레이트를 낮춰 타이밍 마진을 확보했다.

**실측 근거 (우리가 직접 규명한 것)**
- **1 Mbps에서 프레임 드롭 24~52% + bus-off 폭주 → 500 kbps에서 동시 회전 10회 드롭 0%.**
  "긴 케이블은 못 줄이니 비트레이트로 마진을 산다"는 결론을 실험으로 확정.
- **모터 PWM 노이즈의 정체를 정량 규명**: 정지 상태 폐루프 유지 시 CAN 송신 오류 **27.9%**,
  회전 중에는 **약 2%** — 즉 원인은 배선이 아니라 **모터의 전기적 상태**였다. 정지 시 3상 듀티가
  모두 ≈50%로 모여 스위칭 에지가 동시에 터지면서 공통모드 노이즈가 최대가 되고, 회전하면 듀티가
  분산되어 사라진다(SVM 에지 정렬 메커니즘).
- **절연 트랜시버 교체 후 동일 조건에서 노이즈 완전 소멸**: 최악 조건 **27.9% → 0.0%**,
  30초 프레임 폭격 시험 **74.6% → 0.00% (0 / 13,205 프레임)**.

**차별점** — 다축 로봇에서 흔한 "모터군마다 별도 버스/별도 MCU" 구조를 버리고 **단일 채널로 통합**했다.
배선·커넥터·부품 수가 줄어 1차 심사평의 *"부품이 방대하다"* 지적에 구조적으로 대응한다. 또한
"CAN 에러가 난다"에서 멈추지 않고 **에러의 물리적 발생 기전까지 규명해 하드웨어 처방(절연)으로 종결**한
과정 자체가 검증 역량의 근거다(실험 16종, 전말 문서화).

**한계 (명문화)** — 단일 버스는 **배선 단순화의 대가로 단일 장애점(SPOF)** 이다. 버스 1개가 죽으면
10모터 전체가 멈춘다. 이는 의도적 트레이드오프이며, 그래서 ⑥-C의 **워치독 자동복구**와
**US-100 독립 안전 경로**(CAN과 다른 물리 채널인 UART)를 이중 안전망으로 둔다.

### 5-B. 영상과 좌표의 분리 스트리밍 [완료 · 성능 HIL 종결]

**개념** — 원격 운용자가 보는 **영상**과, 자율/정밀 접근에 쓰는 **좌표 데이터**를 **한 스트림에 섞지 않고
분리해 전송**한다. 영상은 SRT(H.264), 좌표는 별도 UDP JSON.

**왜 분리하는가** — 영상에 인식 결과(박스·좌표)를 그려 넣어 보내면, **영상이 YOLO 추론이 끝날 때까지
기다려야 한다** — 추론 지연이 그대로 조종 영상 지연이 된다. 그래서 파이프라인을 **영상 먼저 송신 →
추론은 뒤따라 좌표만 별도 송신**으로 뒤집었다. 조종자는 최저 지연 영상을 보고, 좌표는 필요한 소비자만
구독한다.

**구현** — 젯슨: `vision/gst_stream.py`(공용 송신), `vision/yolo_depth_3d.py`(YOLO + depth → 3D 좌표).
Orin Nano에는 NVENC 하드웨어 인코더가 **없어서** x264 SW 인코딩으로 확정(검증된 제약).
노트북 수신은 **2경로**: ① `scripts/recv_stream.sh` = 오버레이 없는 저지연 네이티브 뷰어(원격주행용),
② `scripts/recv_yolo3d.py` = 좌표 오버레이 뷰어(정밀 접근·좌표 점검용). 지연/용도에 따라 골라 쓴다.

**실측 근거** — L515 게이트웨이 성능 HIL: **RGB SRT 수신 29.91 fps, ROS color 30.0~30.2 Hz,
Depth 10.0 Hz, SRT 드롭 0.**

**한계** — 무선 구간 대역폭에 종속된다(WiFi 절전 모드가 최대 지연 요인이었음 — 끄는 것이 1순위 처방).
현재 기본 해상도 848×480, SRT latency 60 ms는 **지연과 끊김의 트레이드오프 파라미터**로, 현장 전파
환경에 따라 재조정이 필요하다.

### 5-C. 로봇 전용 무선 AP [완료]

**개념** — 로봇에 **전용 WiFi 라우터(GL-SFT1200)** 를 탑재해, 공용망에 의존하지 않는 독립 무선 링크를
구성한다. 젯슨은 **고정 IP 예약**, 조종 노트북은 **5 GHz 대역** 접속.

**왜 전용 AP인가** — 대회장·재난 현장의 공용 WiFi는 혼잡·인증·차단 문제로 신뢰할 수 없다. 전용 AP는
① 채널을 우리가 고르고 ② 접속 기기가 로봇+조종기뿐이라 경합이 없으며 ③ 젯슨 IP가 고정되어
스크립트·스트리밍 주소가 절대 바뀌지 않는다(운용 재현성).

**한계** — WiFi는 본질적으로 **가시선·거리 제약**이 있다. 그래서 통신 두절을 "예외"가 아니라
**정상 시나리오**로 간주하고, 링크가 끊기면 구동 출력을 0으로 떨어뜨리는 워치독을 제어 루프에
내장했다(⑥-A).

### 5-D. 원격 조종 (무선 4WS 텔레옵) [완료 · 실기 검증]

**개념** — DualSense 게임패드 → 노트북 → WiFi → 젯슨 → CAN → 10모터. 조종자는 **바퀴 10개를 개별로
조작하지 않고**, 차체 단위의 직관적 명령(전진/후진, 회전, 제자리 선회)만 준다.

**구조적 선택 — "얇은 클라이언트, 두꺼운 서버"** — 노트북은 **게임패드 raw 입력만** TCP로 던지고
(`"left_x rt lt sq ci\n"`, 30 Hz), **매핑·속도 한계·최저속도 플로어·피벗 판정·안전 게이팅은 전부 로봇
쪽(젯슨)에서** 수행한다. 안전에 관련된 판단을 무선 너머에 두지 않기 위해서다. 링크가 끊기면 서버가
스스로 구동을 0으로 만든다(조종자 개입 불필요).

**실측 근거** — 유선과 **동일 코드 경로**로 무선 엔드투엔드 검증 완료:
- 전진: 구동 4축 **2.40~2.42 rev/s 균일**
- 좌회전: 애커만 차동이 실제로 발생 — **좌측 1.46 < 우측 2.01 rev/s**
- arm(무장) / estop(비상정지) / 링크 끊김 시 정지 동작 확인

**운용 노하우(실기에서 얻은 것)** — 구동 모터의 HALL 센서는 **0.3 rev/s 미만에서 코깅 데드존**에 빠져
텔레메트리는 정상인데 실물이 안 도는 현상이 있다. 그래서 **최저 구동속도 플로어**를 두어 명령이
0이 아닌 이상 데드존 위로 올려 보낸다. 조향 응답성은 원격 조종 감각에 맞춰 슬루 속도를 상향
(출력축 약 47°/s, 45° 전환 0.85초 — 모터 정격 이내).

**한계** — 현재 조종은 **영상 기반 시계 조종**이라 조종자 숙련도와 영상 지연에 성능이 좌우된다.
게임패드 키 매핑은 **HIL·운전자 피드백 후 변경 가능한 초기 후보**로 관리한다(확정값 아님).

### 5-E. 중간 MCU 제거 — 젯슨 직접 제어 [완료 · 구조적 결정]

**개념** — 통상의 로봇은 `젯슨(상위) → 시리얼 → MCU(하위, Arduino/STM32) → 모터드라이버` 구조를 쓴다.
우리는 **중간 MCU를 없애고 젯슨이 socketcan으로 모터를, UART로 안전센서를 직접 잡는다.**

**얻는 것**
1. **지연 감소** — 상위-하위 간 프로토콜 변환·재전송 계층이 통째로 사라진다. 제어 명령이 거치는 홉이 줄어
   50 Hz 제어 루프의 타이밍 예산이 확보된다.
2. **구조 단순화 = 부품·펌웨어 감소** — MCU 보드, 그 전원, 그 케이블, 그 펌웨어(및 펌웨어 업데이트
   절차)가 전부 BOM과 유지보수 목록에서 빠진다. 1차 심사평 *"부품이 방대하다"* 에 대한 직접적 대응.
3. **디버깅 단일화** — 문제가 나면 볼 곳이 한 군데(젯슨)다. "MCU 펌웨어인가 상위인가"라는 흔한
   이중 진단 비용이 없다.

**대가(정직하게)** — 실시간성을 리눅스 유저스페이스가 떠안는다. 그래서 제어 루프를 **논블로킹 50 Hz**로
설계하고 실제로 주기를 측정해 근거를 남겼다(⑥-A의 p99 0.280 ms / overrun 0). 또한 CAN 드라이버 계층
장애(TX 웻지)를 **소프트웨어 워치독으로 자가복구**하도록 만들어, MCU가 해주던 "죽으면 리셋" 역할을
대체했다(⑥-C).

### 5-F. 통신·모터 진단 GUI [완료]

**개념** — 브라우저에서 접속하는 **웹 기반 모터 진단·튜닝 GUI**(`motor_gui`, FastAPI + WebSocket).
100 Hz로 모터 상태를 샘플링해 실시간 그래프로 보여주고, CSV/Parquet로 기록한다.

**핵심 설계 — 트랜스포트 추상화** — GUI 내부는 `Transport` / `CanDevice` 추상 인터페이스로 짜여 있어,
**AK 조향 모터와 ODrive 구동 모터를 "컴포저블 디바이스"로 묶어 하나의 CAN 버스에서 동시에** 다룬다.
새 장치가 생기면 인터페이스만 구현해 드롭인하면 된다(⑦-F와 같은 패턴).

**영상 GUI 경로** — 스트리밍 수신은 GStreamer 기반 뷰어 2종(5-B 참조)으로 분리 운용한다.
진단 GUI(모터 텔레메트리)와 조종 영상(저지연 뷰어)을 **의도적으로 분리**해, 무거운 시각화가 조종
영상 지연에 끼어들지 않게 했다.

**한계** — 진단 GUI는 **개발·정비용**이다. 대회 실주행 중에는 CPU/네트워크 예산을 조종 영상에
몰아주기 위해 상시 구동을 전제하지 않는다.

## 6. 안전 · 신뢰성

> 설계 철학: **안전은 "잘 동작할 때"가 아니라 "무엇이 고장 났을 때"를 기준으로 설계한다.**
> 그래서 안전 판정 로직은 ROS·CAN·영상 어디에도 의존하지 않는 **순수 파이썬 코어**로 분리했고,
> 상위 소프트웨어가 통째로 죽어도 정지가 성립하도록 경로를 나눴다.

### 6-A. 비상 정지 로직 — 독립 경로 정지 + 복귀 로직 [완료 · 실기 HIL]

**독립 경로** — 충돌 방지 센서(US-100)는 **CAN이 아닌 UART**(`/dev/ttyTHS1`)에 물려 있다. 즉 모터를
제어하는 통신 채널과 **물리적으로 다른 채널**로 정지 신호가 올라온다. CAN 버스가 통째로 죽어도 안전
판정은 살아 있고, 반대로 센서가 죽어도 판정은 **정지 쪽으로 실패(fail-safe)** 한다.

**정지의 종류를 구분한 것이 핵심** — 모든 이상을 똑같이 "정지"로 묶으면, 사소한 노이즈에도 사람이
가서 리셋해야 해서 운용이 불가능해진다. 그래서 두 등급으로 나눴다.

| 등급 | 발동 조건 | 복귀 |
|---|---|---|
| **ESTOP (래치)** | 유효한 근접 거리 검출, 또는 센서 응답 3회 연속 실패 | **자동 복귀 없음.** 사람이 명시적으로 리셋 → `IDLE` 복귀. **리셋이 곧 재출발은 아니며, 무장(arm)은 별도 동작**이다. |
| **MOTION_HOLD (자동복구)** | 일시적 미확정 판정, `/cmd_vel` 명령 두절(0.5 s), 순간 통신 끊김 | 조건이 해소되면 **자동 복귀** |

이 "래치 vs 자동복구" 분리와 **"리셋 ≠ 자동 재출발"** 규칙이, 사고 후 로봇이 스스로 다시 움직이는
최악의 시나리오를 구조적으로 차단한다.

**실측 근거 (60초 연속 HIL)**
- 제어 루프: **3000틱 / 60초, 5초 창 평균·최저 모두 50.000 Hz, 틱 p99 0.280 ms, overrun 0**
- **상위 노드가 죽으면 0.753초 만에 자동 E-stop** (통신 두절 → 정지 성립)
- 시작 시 `ESTOP` → 안전 거리 확보 시 `ARMED/RUN` → 근접 시 `ESTOP` → 멀어져도 **래치 유지** →
  리셋 시 `IDLE`(자동 무장 없음) 전 과정 확인
- 회귀 테스트: `motor_control` 198 passed, `motor_gui` 91 passed, `powertrain_ros` 32/32

**미결(정직하게)** — 최종 **정지 거리(stop_mm)** 는 **실차 조립 후 지상 제동 시험으로 확정**한다.
현재 200 mm는 벤치/HIL 값이며, 50 kg 실차의 관성·노면 마찰을 반영한 값이 아니다. **경사로 정지 시
미끄러짐** 대응(정지 상태 유지 토크)도 실차 커미셔닝 항목이다.

### 6-B. 물리적 긴급 정지 [HW 소관 · SW 연계 설계 완료]

**개념** — 소프트웨어와 **완전히 독립된 하드웨어 킬 스위치**로 모터 전원을 차단한다. SW가 어떤 상태든
(커널 패닉·프로세스 행 포함) 사람이 손으로 끊을 수 있는 최후 수단이다.

**SW 측 대응(우리 몫)** — 전원이 복구되어 다시 켜졌을 때 로봇이 **곧바로 움직이지 않도록**, 제어
소프트웨어는 **시작 상태를 항상 `ESTOP`으로 두고 부팅**한다(무장은 명시적 동작). 즉 물리 정지 후
전원 복구가 곧 재출발이 되는 경로를 SW가 차단한다. 또한 구동 모터의 캘리브레이션은 RAM에만 있어
**전원 차단 시 소실**되므로, 재기동 절차(재캘리브레이션)를 운용 문서에 명시했다.

> ⚠️ 하드웨어 킬 스위치의 회로·장착은 **전장(파워) 소관**이다. 본 SW 문서는 "SW가 요구하는 조건"만
> 명시한다: ① 모터 전원 차단은 젯슨 전원과 분리될 것(로그·상태 보존), ② 차단 상태를 SW가 읽을 수
> 있으면 더 좋음(현재는 미가정).

### 6-C. 워치독 — CAN 자동 복구 [완료 · 상주 서비스]

**해결한 문제** — 모터 PWM 노이즈로 bus-off가 반복되면, 젯슨의 CAN 드라이버(mttcan)가
**송신 큐를 영구히 멈추는 커널 레벨 장애("TX 웻지")** 에 빠진다. 겉으로는 에러 카운터가 0이고
링크도 UP인데, **모든 송신이 실패**한다 — "잘 되다가 갑자기 완전 먹통"의 정체가 이것이었다.
드라이버 재초기화(`down/up`) 외에는 복구 수단이 없다.

**구현** — `corner_module/can_watchdog.py`: 주기적으로 프로브 프레임을 쏘고 커널의 송신 카운터가
멈췄는지 감시하다가, 웻지를 감지하면 **인터페이스를 자동으로 내렸다 올린다**. 파이썬 표준
라이브러리(ioctl)만 사용해 CAN 라이브러리 장애와 독립적으로 동작한다.

**3중 배치** — ① **컨테이너 상주 서비스**(도커 스택이 뜨면 같이 뜸, `restart: unless-stopped`),
② 텔레옵 프로세스 **인프로세스 내장**, ③ 호스트용 bash 스크립트(비상용).

**실측 근거** — 강제 재현(45초 프레임 폭격 → 송신 0/30 완전 정지) 후 **자동 down/up으로 30/30 복구**.
감지 시간 **약 2초, 오탐 0회**.

**현재 위상(정직하게)** — 절연 트랜시버 교체로 **웻지의 근본 원인은 제거**되었다(6-A·5-A 참조).
워치독은 제거하지 않고 **보험으로 상주**시킨다 — 원인을 없앴다고 해서 커널 장애 가능성이 0이 되는 것은
아니며, 대회 현장에서 재부팅으로 대응할 시간이 없기 때문이다.

## 7. SW / 자율주행

> **스코프 정직성**: 아래 7-D(L515), 7-E(Rviz), 7-F(4WS)는 **완료·검증**된 항목이고,
> 7-A·7-B(자율주행 본체·주행 FSM)는 **설계 완료 + 구현 진행/예정** 단계다.
> 로봇팔팀과 동일하게 "됐다"가 아니라 "어디까지 됐고 무엇이 남았는가"로 쓴다.

### 7-F. 4WS 협조 조향 — 트랜스포트 추상화 [완료 · 10모터 실기 HIL]

> (구현 성숙도 순으로 F부터 서술한다. 자율주행의 **실행 계층**이 여기다.)

**개념** — 상위(조종기든 자율주행이든)는 **차체 명령 두 개, 즉 전진속도 v 와 회전속도 ω 만** 준다.
`ChassisManager`가 이를 **애커만 기하학**으로 풀어 **조향 4축의 각도 + 구동 6축의 속도**로 분배한다.
안쪽 바퀴와 바깥쪽 바퀴가 서로 다른 속도로 돌아야 미끄러짐 없이 도는데(차동), 그 계산이 이 계층에 있다.

**트랜스포트 추상화 — 이 항목의 핵심 차별점** — 제어 로직은 **모터가 무엇인지, 어떤 선으로 연결됐는지
전혀 모른다.** `Actuator` / `SteerActuator` / `DriveActuator` 라는 추상 인터페이스만 호출한다.
그 뒤에 무엇이 오든 상관없다:

| 바뀌는 것 | 상위 제어 코드 변경량 |
|---|---|
| ODrive를 **USB → CAN** 으로 전환 | **0줄** (드라이버 교체만) |
| 조향 모터를 **AK40-10 → AK45-36** 으로 교체 | **0줄** (프로파일 전환) |
| 실제 모터 대신 **가짜 드라이버로 테스트** | **0줄** (fake 주입) |

덕분에 **하드웨어 없이 전체 제어 로직을 단위 테스트**할 수 있고(실기 없이 회귀 검증 가능),
모터가 바뀌어도 검증된 상위 로직을 버리지 않는다. 이것이 로봇팔팀의
"엔드이펙터가 바뀌어도 인식·접근 인프라를 재사용"하는 것과 **동일한 범용화 패턴**이다 —
**팔은 손끝만 교체, 차체는 드라이버만 교체.**

**안전 통합** — `ChassisManager`는 비상정지를 6개 코너 모듈 전체에 전파하고, US-100 판정으로 구동을
게이팅하며, 차체 워치독을 돌린다. 즉 **안전이 상위 앱이 아니라 차체 계층에 내장**되어 있어, 어떤
상위(수동 조종/자율주행)를 붙여도 안전 거동이 동일하다.

**실측 근거** — 10모터(조향 4 + 구동 6) 단일 CAN 협조 제어 실기 HIL 완주:
조향 원점 → 전진 → 좌/우 선회(애커만 차동 + 역위상 4WS, 조향 꺾임과 바퀴 회전 동시 육안 확인) → 정지.
`/cmd_vel` 명령 경로 HIL에서 **전진 시 구동축 1.71~1.88 rev/s 균일**, 명령 두절 시 워치독 정지 확인.

**한계** — 저속(<0.3 rev/s) HALL 코깅 데드존이 물리적으로 존재해, 최저 구동속도 플로어로 우회한다.
초저속 정밀 접근(팔 작업 직전 미세 정렬)은 이 플로어 때문에 제약이 있으며, 필요 시 별도 저속 모드가
필요하다 — **로봇팔 정밀 접근과 맞물리는 미결 항목**.

### 7-D. LiDAR(L515) — pyrealsense2 기반 거리 확보 [완료 · 성능 HIL 종결]

**개념** — Intel RealSense **L515는 솔리드스테이트 LiDAR**로, `pyrealsense2` SDK로 직접 읽어
**RGB 영상 + 픽셀별 실거리(depth) + IMU**를 한 센서에서 얻는다. 스테레오 추정이 아닌 ToF 실측이라
거리값이 정확하다.

**센서 역할 분리(팀 합의 완료)** — **L515 = 파워트레인(주행) 전용 / D435i = 로봇팔 인식 전용 /
US-100 = 독립 안전.** 초기에 로봇팔이 D435i를 독점해 우리가 쓸 카메라가 없던 블로커를
**센서 분리로 해소**했다. 파워트레인은 로봇팔의 원본 영상을 점유하지 않고, 표준화된 인식 결과
토픽(`/detected_objects`)만 구독한다 — 로봇팔팀의 "단일 인식원" 설계를 우리도 그대로 따른다.

**실측 근거** — 게이트웨이 성능 HIL: **ROS color 30.0~30.2 Hz, Depth 10.0 Hz, SRT 수신 29.91 fps,
드롭 0.** 센서 분리/재연결, D435i와 동시 60초 연결 안정성 확인.

**한계** — L515는 **강한 직사광에서 ToF 성능이 저하**되는 특성이 있다(실외 시나리오 리스크로 명문화).
현재 파이프라인은 **경량 color + depth + IMU**이며, PointCloud2는 선택 항목이다.

### 7-E. Rviz 시각화 인터페이스 [완료 · 제어 경로와 분리]

**개념** — 로봇 URDF(차체 모델), 실시간 바퀴/조향 상태, L515 포인트클라우드, 지형 기울기, 장애물
구역 판정을 **Rviz에서 한 화면에 겹쳐 본다.** 조종자·개발자가 "로봇이 지금 무엇을 보고 무엇으로
판단했는가"를 눈으로 확인하는 계층이다.

**설계 원칙 — 시각화는 제어를 방해하지 않는다** — 시각화 노드(포인트클라우드·URDF·기울기)는
**프로덕션 제어 경로에서 격리**되어 있다. 시각화를 끄든 켜든 50 Hz 제어 루프의 거동은 동일하다.
디버깅 도구가 실주행 성능을 갉아먹는 흔한 함정을 구조로 차단했다.

**현재 상태** — 구현 완료(URDF·조인트 브리지·클라우드·기울기·장애물 구역 노드), **실주행 튜닝은
지형 시험과 함께 진행 예정**.

### 7-A. 자율 주행 [설계 완료 · 구현 진행 중]

**목표** — 원격 조종 없이 지정 경로(레인)를 따라 주행하고, 장애물을 회피하며, 로봇팔 작업 지점에
정차한 뒤, 작업 완료 신호를 받아 재출발한다.

**현재까지 완료된 하부 구조** (자율주행은 아래 위에 올라간다)
- ✅ 차체 실행 계층(4WS, 10모터) — 7-F
- ✅ 안전 계층(독립 경로 정지·워치독) — 6장
- ✅ 인지 입력(L515 RGB/depth/IMU + 로봇팔 인식 결과 구독) — 7-D
- ✅ 로봇팔과의 **ROS2 메시지 왕복 통신 검증**(양방향 DDS 전달 확인)
- 🔄 **오도메트리(자기 위치 추정)** — 바퀴 회전 + IMU 융합으로 `/odom` 발행. 구현·단위테스트 완료,
  **실주행 정확도 검증 진행 중**
- 📋 경로 추종·미션 시퀀서 — **예정**

**로봇팔과의 핸드셰이크(크로스팀 미결)** — `도착 알림 → 팔 작업 → 완료(DONE) → 재출발`의
1사이클 합동 검증과, 정차 중 자세 잠금 해제 순서 계약이 **양 팀 공통 미결 항목**이다.
(로봇팔팀 문서의 "status enum·토픽 스펙 최종 합의"와 **같은 항목** — 우리 쪽에서도 최우선.)

### 7-B. 주행 알고리즘 + FSM [설계 완료 · 구현 예정]

**개념** — 주행을 **3계층 + 행동 상태기계**로 구성한다. 로봇팔의
`PERCEIVING → PLANNING → EXECUTING → DONE`과 **같은 철학** — 공통 뼈대는 고정하고,
지형·미션별로 **행동(behavior)만 교체**한다.

| 계층 | 역할 | 상태 |
|---|---|---|
| **미션 계층** | 대회 시나리오 순서(지형 → 작업 지점 → 다음 지형) 관리 | 설계 완료 |
| **행동 계층** | 지형별 주행 행동(레인 추종 / 계단 / 경사 / 정밀 접근 / 정차 / 대기 등) | 설계 완료, 구현 예정 |
| **실행 계층** | 4WS 차체 명령(v, ω) 하달 — 7-F | **완료** |

**차별점** — 행동을 교체 가능한 단위로 쪼갰기 때문에, **지형이 추가되면 행동 하나만 추가**하면 된다.
인지·차체·안전 인프라는 그대로 재사용한다(= 로봇팔의 "EXECUTING 상태만 교체"와 동일 패턴).

**한계** — 지형별 행동의 **파라미터(속도·조향 게인·통과 판정)는 실지형 시험 없이는 확정 불가**다.
현재는 구조만 확정했고, 수치는 지형 시험에서 채운다.

### 7-C. 비전 · 안전 융합 [부분 완료]

**개념** — **서로 다른 원리의 센서를 계층으로 쌓아** 한 센서의 실패가 곧 사고가 되지 않게 한다.

| 계층 | 센서 | 원리 | 역할 | 상태 |
|---|---|---|---|---|
| **최후 방어선** | US-100 초음파 | 음파 ToF | 근접 시 무조건 정지. **CAN과 다른 채널(UART)**, 순수 파이썬 판정 | ✅ **완료·HIL** |
| **주행 판단** | L515 (LiDAR) | 광 ToF | 전방 지형/장애물 거리, 좌·중·우 구역 판정 | 🔄 프로토타입 |
| **의미 인식** | 로봇팔 YOLO | 학습 기반 | 물체 종류·위치(`/detected_objects` 구독) | 🔄 필터링 예정 |

**융합 원칙 — "위 계층이 틀려도 아래 계층이 막는다"** — YOLO가 오인식하거나(닫힌 집합의 한계),
L515가 직사광에 흔들려도, **초음파 정지는 학습 모델도 영상도 CAN도 거치지 않는다.** 딥러닝의
불확실성을 안전 경로에서 완전히 배제한 것이 이 구조의 요지다.

**한계(정직하게)** — 초음파는 **단일 센서·전방 한정**이라 측면·낙하(cliff) 위험은 커버하지 않는다
(현 스코프에서 의도적 제외). L515 기반 장애물 구역 판정은 **아직 시각화·프로토타입 수준이며 제어
루프에 물려 있지 않다** — 실지형 검증 후 편입 예정.

## 요약 블록 (로봇팔팀 문서와 동일 형식)

### 1. 창의성

**① CAN 에러의 물리적 기전 규명 → 하드웨어 처방으로 종결**
"에러가 나니 재시도한다"가 아니라, **정지 시 27.9% vs 회전 시 2%** 라는 반직관적 실측에서 출발해
"모터 정지 상태에서 3상 스위칭 에지가 동시에 터진다"는 기전까지 규명하고, **절연으로 원천 차단**
(27.9% → 0.0%). 실험 16종으로 문서화. — *증상 대응이 아니라 원인 규명*

**② 안전을 상위 소프트웨어에서 분리한 독립 경로 설계**
안전 판정을 ROS·CAN·영상 어디에도 의존하지 않는 **순수 파이썬 코어 + 별도 물리 채널(UART)** 로 분리.
**상위 노드가 죽으면 0.753초 만에 자동 정지**함을 실측. 딥러닝·영상·CAN이 전부 실패해도 정지가 성립한다.

**③ "래치 vs 자동복구" 정지 등급 분리 + 리셋 ≠ 재출발**
모든 이상을 똑같이 정지시키면 운용이 불가능하고, 전부 자동복구하면 위험하다. **위험은 래치하고
일시 장애는 자동 복구**하되, **리셋해도 자동으로 무장되지 않게** 만들어 사고 후 자기 재출발을 구조적으로 차단.

**④ 트랜스포트 추상화 — 하드웨어가 바뀌어도 검증된 로직을 버리지 않는다**
USB↔CAN 전환, 모터 모델 교체, 가짜 드라이버 테스트가 **상위 제어 코드 0줄 수정**으로 성립.
로봇팔의 "엔드이펙터만 교체" 패턴과 동일한 범용화 원리를 **차체에 적용**했다.

**⑤ 영상과 좌표의 분리 전송**
인식 결과를 영상에 그려 보내면 조종 영상이 추론 지연만큼 늦어진다. **영상을 추론보다 먼저 내보내고
좌표는 별도 채널로** 보내, 조종 지연과 인식 정확도를 서로 인질로 잡지 않게 했다.

**⑥ 중간 MCU 제거**
젯슨이 CAN·UART를 직접 잡아 **홉·부품·펌웨어·이중 디버깅 비용을 동시에 제거**. 그 대가인 리눅스
실시간성 리스크는 **50 Hz 주기 실측(p99 0.280 ms, overrun 0)** 과 **커널 CAN 장애 자동복구 워치독**으로 상쇄.

### 2. 목적

- **모듈형 로봇을 뒷받침하는 범용 주행 소프트웨어 구축**: 모터·연결·지형이 바뀌어도 재사용되는
  차체·안전 인프라 확보(하드웨어 종속 제거).
- **1차 심사평 대응**
  - *부품이 방대하다* → **단일 CAN 채널 통합 + 중간 MCU 제거 + 카메라 역할 분리**로 실제 부품 수를 줄임
  - *검증 기준이 없다* → **"됐다" 대신 "무엇을 몇 회 측정해 몇 이하이면 합격"** 을 아래 표로 제시
  - *비용* → 저가 SBC(Jetson Orin Nano) 한 대로 인지·제어·통신·안전을 모두 수행

### 3. 기대효과

- **확장성**: 지형/미션 추가 = 행동(behavior) 하나 추가. 모터 교체 = 드라이버 하나 교체.
  인지·차체·안전 인프라 100% 재사용.
- **경제성(BOM 절감)**: 하위 MCU 보드·별도 버스·별도 배선 제거, 고가 GPU 없이 Orin Nano로 구동.
- **신뢰성**: 통신 두절·센서 고장·커널 장애 각각에 대해 **독립적인 정지·복구 경로**를 확보 →
  재난 현장처럼 통신이 불안정한 환경에 그대로 적용 가능.
- **발표 신뢰도**: 완료/진행/예정을 투명하게 구분하고 **실측 수치로만 완료를 주장** → "검증 기준 부재"
  지적을 정면 반박.

## 한계점 요약 (파워트레인 SW)

1. **단일 CAN 버스 = 단일 장애점**. 배선 단순화의 의도적 대가. 워치독 자동복구 + 독립 채널 안전(US-100)로 완화하나, 버스 물리 단선은 커버하지 못한다.
2. **최종 정지거리(stop_mm) 미확정**. 현재 200 mm는 벤치 값. 50 kg 실차의 지상 제동거리는 **차체 조립 후 커미셔닝**에서 실측해야 한다. **경사로 정지 시 미끄러짐**도 동일 단계 항목.
3. **초음파 안전은 단일·전방 한정**. 측면 충돌·낙하(cliff)는 현 스코프 제외.
4. **저속 데드존(HALL 코깅 <0.3 rev/s)**. 최저속도 플로어로 우회 중 — 팔 작업 직전 초저속 정밀 정렬에 제약이 있어, 로봇팔팀과 맞물리는 미결 항목.
5. **자율주행 본체는 미완**. 오도메트리는 실주행 정확도 검증 중, 경로 추종·미션 시퀀서는 예정.
6. **L515 직사광 취약**. ToF 원리상 실외 강광에서 depth 품질 저하 — 실외 시나리오 리스크로 명문화.
7. **크로스팀 미결**: 로봇팔과의 `도착 → 작업 → DONE → 재출발` 1사이클 합동 검증 및 status enum·토픽 스펙 최종 합의. (기술 문제가 아니라 합의 문제 — 양 팀 공통 최우선)
8. **원격 조종은 시계 조종**. 조종자 숙련도·영상 지연에 성능이 좌우된다. 게임패드 키 매핑은 확정값이 아닌 초기 후보.
