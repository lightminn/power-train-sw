# WP5.2 Robot-Arm Collaboration Safety Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use
> checkbox (`- [ ]`) syntax for tracking.

**Goal:** 기존 `robot_arm_msgs` 5종을 유지하면서 주행 중 팔 잠금, 작업 중 차체 정지,
실물 CAN 단일 소유와 실패 후 안전 복구를 fail-closed로 구현한다.

**Architecture:** ROS 없는 `ArmInterlock`이 status·stamp·mission correlation을 판정하고,
`chassis_node`가 이를 `SafetyInterlock`의 motion hold로 최종 집행한다. 실물 CAN owner는
별도 `flock` helper로 entry point 전체에서 강제한다. 로봇팔 저장소는 별도 팀 소유이므로
계약시험과 동시 컷오버 요구사항만 이 저장소에 고정한다.

**Tech Stack:** Python 3.10, ROS2 Humble/rclpy, `robot_arm_msgs`, Linux `fcntl.flock`, pytest,
Docker Compose, Jetson Orin Nano.

## Global Constraints

- 대회 범위는 2026 국방로봇경진대회뿐이다.
- `robot_arm_msgs` 5종의 wire schema와 토픽명은 변경하지 않는다.
- DualSense 원격 입력은 5종 협업 msg를 확장하지 않고 별도 versioned operator-input transport와
  표준 `control_msgs/msg/JointJog`를 사용한다. 노트북은 DDS participant가 되지 않고 Jetson의 단일
  `remote_input_gateway`만 입력을 수신한다.
- 현행 구 로봇팔과 새 chassis 혼합 버전은 `contract_v2_verified=false` compatibility-lock profile로만
  기동한다. 이 profile은 구 팔도 잠그는 wire value `CORNERING`만 10 Hz로 발행하고 `DRIVING`,
  `MISSION_STOP`, `ArrivalStatus`는 발행하지 않아 nonzero 구동과 취소 모션 재개를 함께 차단한다.
- `MISSION_STOP`만 작업 허가이며 `STOW_REQUEST`는 접힘·잠금 요청이다.
- `DRIVING/CORNERING/ROUGH_TERRAIN/FOLLOW_LEAD`, 미인식 mode와 stale은 모두 잠금 유지다.
- heartbeat는 header stamp 기준 10 Hz, timeout 0.5 s, Reliable/Keep Last 1/Volatile이다.
- freshness 식은 `0 ≤ now-stamp ≤ timeout`이다. node clock rollback·clock-domain 변경은 기존 sample과
  dwell을 전부 무효화하고 새 단조 heartbeat가 쌓일 때까지 hold한다.
- 팔 통신 장애와 `GRIP_LOST`는 E-stop이 아니라 motion hold다. `GRIP_LOST`는 supervisor latch다.
- base 계약의 필수 실패 상태는 현행 wire 어휘 `FAILED`다. `IK_FAILURE`, `TRAJECTORY_FAILURE`,
  `SELF_COLLISION`, `BASE_COLLISION`, `JOINT_OVERCURRENT`, `GRIP_UNCERTAIN`, `STOW_FAILURE`,
  `ACTION_TIMEOUT`은 로봇팔 팀이 합의·구현할 경우에만 추가하는 선택적 진단 어휘이며 안전 gate의
  필수 의존성이 아니다. 어떤 실패 값도 주행 성공 ACK로 사용하지 않는다.
- E-stop reset은 `IDLE`까지만 복구하며 별도 arm이 필요하다는 기존 계약을 바꾸지 않는다.
- 실물 motor/CAN 검증은 Jetson `powertrain_jetson` 또는 `powertrain_ros` 컨테이너에서 수행한다.
- 로봇팔 Jetson checkout의 기존 modified/untracked 파일을 보존하고 이 계획에서 직접 수정하지 않는다.
- 정지 판정은 코드 상수로 고정하지 않는다. ODrive 11~16 모두에서 전진·후진·좌/우 pivot 뒤 정지
  노이즈를 측정해 per-wheel threshold와 dwell을 production YAML에 동결한다. node 12/16 HALL 보강과
  node 13/14 포함 6축 qualification 전에는 실물 팔 작업 허가를 production으로 승인하지 않는다.
- 구현 순서는 Task 1 → 2 → 3 → 4 → 5 → 6 → 7이다. 뒤 Task가 만드는 파일을 앞 Task가 참조하지
  않으며 Observability 계획은 Task 1·3·4·5 완료 뒤에만 시작한다.
- 파워트레인 단독 실차 시험용 `arm_gate_mode=arm_absent_field` launch profile을 Task 2에서 함께
  구현한다. 활성 조건: ① `/arm_status` publisher가 ROS graph에 부재(주기 재확인, publisher 출현
  시 다음 tick에 mock 비활성·default-deny 복귀), ② 운영자가 팔의 기계적 접힘·고정을 육안
  확인(바퀴 부양 확인과 동급의 명시 절차), ③ journal/로그에 profile·확인자 기록. 이 profile은
  내부 mock locked heartbeat로 arm freshness gate만 대체하며 `MISSION_STOP`/`ArrivalStatus`
  발행과 팔 작업 허가는 계속 금지되고 US-100·motor·E-stop·command watchdog·authority gate는
  그대로다. 대회 production launch에는 포함하지 않는다. 이 profile이 없으면 팔 팀 v2 배포 전까지
  모든 실차 주행 HIL이 차단된다는 것이 도입 근거다.
- Verify 공통 규칙: 셸 체인은 `set -e`로 실패를 전파한다(pytest가 마지막 명령이 아닌 체인에서
  `;`가 실패를 은폐하지 않게). `chassis_node` 등 generated msg를 import하는 시험은 pytest 전에
  `cd ros2 && colcon build --packages-select robot_arm_msgs powertrain_msgs --build-base /tmp/build
  --install-base /tmp/install`(read-only 마운트이므로 /tmp에 빌드) 후 `source /tmp/install/setup.bash`를
  수행한다. 순수 코어 시험(Task 1 등)은 이 빌드가 필요 없다.
- production process는 `powertrain_ros`(L515 Gateway), `powertrain_control`(chassis/US-100/teleop
  adapter), `powertrain_autonomy`, `powertrain_observability`로 분리한다. control과 camera ownership을
  같은 Compose service에 섞지 않는다.
- D435i는 로봇팔 camera-owner process 하나만 SDK를 연다. 원격 팔 영상은 raw RGB
  848×480×30 SRT `:5002`, YOLO metadata UDP `:5003`으로 분리하며, 현행
  `/perception/debug_image → SRT :5000` 시험 경로는 production에서 사용하지 않는다. L515
  1280×720×30 SRT `:5000`과 동시에 유지하고 양 sender가 서로를 재시작하지 않는다.
- DualSense `DRIVE`와 `ARM` mode는 상호배타다. Jetson authority ACK 전에는 mode가 바뀌지 않으며,
  한 frame이나 한 control tick에서 nonzero chassis command와 nonzero arm jog를 함께 만들지 않는다.

- 팔 측 locked adapter는 승인 pose tolerance, 모든 joint velocity threshold, dwell, action/cancel

브랜치 `arm_fsm_node.py`(887행)에서 read-only로 재검증했다.

**코드로 확인된 구현 — 수락:**

- `DRIVING` 자동 언락 분기 삭제, `MISSION_STOP`+동일 `mission_id` `ArrivalStatus` conjunction만
  언락(326행). 미인식 mode 무시(default-deny). `chassis_mode_timeout` 수신 워치독 1.0 s(414행) —
  끊기면 강제 잠금.
- 신규 status 5종 존재. `IDLE`은 `WORK_READY` 대신 `STOWED_LOCKED`를 상시 발행(466행).
  `DONE`은 발행하지 않음 — 성공 권위는 동일 ID locked heartbeat(본 계획과 합치. `DONE` 관련
  시험은 "미래에 수신돼도 무시" 회귀로 유지한다).
- `WORK_READY`는 conjunction 수락 순간의 1회성 ack(368·374행) — **해석 승인**: 파워트레인
  `WORK_ACCEPTED_STATUSES`는 이후의 `PERCEIVING/PLANNING/EXECUTING` 10 Hz 현재상태도 ACK로
  인정하므로 1회성 발행과 호환된다.
- `_publish_status`는 모든 heartbeat에 `header.stamp`·`self.mission_id`를 채운다(866행).
  `mission_id`는 arrival 수락 시 갱신되고 완료 후에도 유지 — 완료 직후 최초 locked heartbeat가
  작업 ID를 실으므로 완료 ACK가 성립한다. 빈손 상시 heartbeat의 ID는 0이 아니라 **마지막 완료
  ID 유지** — 본 계획의 "빈손=`mission_id=0`" 규약을 "0 또는 마지막 완료 ID 유지 모두 유효,
  파워트레인은 완료 ACK 판정에서만 ID를 해석하고 상시 heartbeat ID는 해석하지 않음"으로 완화한다.
- locked 발행 전 확인 `_is_settled`(424행): TF tip 연속 tick 변위 ≤ `locked_pos_tol`(0.005 m),
  관절각 유한차분 속도 ≤ `locked_vel_tol`(0.05 rad/s), dwell 0.5 s, TF 실패 시 dwell 리셋
  (fail-safe), 미충족 시 `EXECUTING` 발행 — locked adapter **1단계**(자세 안정+속도+dwell)에 부합.
- 멱등성: `_last_completed_mission_id`의 arrival 재발행 무시(364행).

**코드로 확인된 잔여 결함 — 회의 안건(팔 팀 deliverable):**

1. **STOW_REQUEST가 payload-aware가 아님(S1)**: `STOW_ABORTABLE_STATES`에 `CARRY`·`LOCKED`가
   포함되고 핸들러가 무조건 `RELEASE`로 전이한다(163~166·333~338행) — 운반 중 수신 시 화물을
   낙하시킨다. 계약 요구: 파지 확정 이후의 `STOW_REQUEST`는 release 없이 `CARRYING_LOCKED`로
   접힘이며, release가 필요한 중단은 별도 명시 경로다.
2. **locked 확인이 정지 안정성만 검사(S1)**: `_is_settled`는 승인 접힘/운반 자세와의 근접성을
   확인하지 않는다 — 임의 자세로 정지한 팔도 0.5 s 뒤 `STOWED_LOCKED`를 발행할 수 있고,
   `STOWING`의 실제 접힘 모션도 TODO 스켈레톤이다(24행 주석). 계약의 "승인 joint pose
   tolerance"는 **승인 자세 근접**을 의미한다. 합동 실차 주행 전 approved-pose proximity 검사와
   실제 stow 모션이 필요하다.
3. controller fault 확인 미포함(브릿지에 필드 부재 — 팔 팀 인지, 2단계 후속). torque hold도 2단계.
4. 수치(pose 0.005 m·vel 0.05 rad/s·dwell 0.5 s·timeout 1.0 s)는 provisional로 수용하고 Task 7
   바퀴 부양 합동 HIL에서 실측·동결한다. `chassis_mode_timeout` 1.0 s는 우리 10 Hz 발행 기준
   수용 범위다.

```python
from dataclasses import dataclass
from math import isfinite

from powertrain_ros import contract

@dataclass(frozen=True)
class ArmSnapshot:
    status: str
    mission_id: int
    stamp_s: float

class ArmInterlock:
    def __init__(self, timeout_s=0.5, future_tolerance_s=0.1):
        self.timeout_s = float(timeout_s)
        self.future_tolerance_s = float(future_tolerance_s)
        self._sample = None
        self._last_seen_stamp_s = 0.0
        self._last_now_s = 0.0
        self._grip_lost_latched = False
        self._contract_violation = None

    def update(self, status, mission_id, stamp_s, now_s):
        if now_s < self._last_now_s:
            self._sample = None
            self._last_seen_stamp_s = 0.0
            self._last_now_s = float(now_s)
            return False
        self._last_now_s = float(now_s)
        if not isfinite(stamp_s):
            return False
        if stamp_s <= 0.0 or stamp_s > now_s + self.future_tolerance_s:
            return False
        if stamp_s <= self._last_seen_stamp_s:
            return False
        self._last_seen_stamp_s = float(stamp_s)
        if status not in contract.ARM_STATUSES:
            self._contract_violation = str(status)
            return False
        self._contract_violation = None
        self._sample = ArmSnapshot(status, int(mission_id), float(stamp_s))
        if status == contract.ARM_GRIP_LOST:
            self._grip_lost_latched = True
        return True

    def fresh(self, now_s):
        if now_s < self._last_now_s:
            self._sample = None
            self._last_seen_stamp_s = 0.0
            self._last_now_s = float(now_s)
            return False
        self._last_now_s = float(now_s)
        age = now_s - self._sample.stamp_s if self._sample is not None else float("inf")
        return 0.0 <= age <= self.timeout_s

    def drive_allowed(self, profile, now_s, manual_override=False):
        if self._grip_lost_latched or self._contract_violation is not None:
            return False
        if manual_override:
            return profile == "REMOTE_ARM_OVERRIDE" and not self.fresh(now_s)
        if not self.fresh(now_s):
            return False
        if profile == "EMPTY_STOWED":
            expected = contract.ARM_STOWED_LOCKED
        elif profile == "CARRYING_LOCKED":
            expected = contract.ARM_CARRYING_LOCKED
        else:
            return False
        return self._sample.status == expected

    def work_acknowledged(self, mission_id, now_s):
        return (self._contract_violation is None
                and self.fresh(now_s)
                and self._sample.status in contract.WORK_ACCEPTED_STATUSES
                and self._sample.mission_id == int(mission_id))

    def clear_grip_lost(self, authorized=False):
        if not authorized:
            return False
        self._grip_lost_latched = False
        return True

    @property
    def last_contract_violation(self):
        return self._contract_violation

    def hold_reason(self, profile, now_s, manual_override=False):
        if self._grip_lost_latched:
            return "grip_lost_latched"
        if self._contract_violation is not None:
            return f"arm_contract_violation:{self._contract_violation}"
        if manual_override:
            if profile != "REMOTE_ARM_OVERRIDE":
                return "operator_override_profile_invalid"
            if self.fresh(now_s):
                return "operator_override_inhibited_by_fresh_arm"
            return ""
        if not self.fresh(now_s):
            return "arm_status_stale"
        if not self.drive_allowed(profile, now_s, manual_override=False):
            return "arm_not_drive_ready"
        return ""

    def operator_status(self, profile, now_s, manual_override=False):
        if manual_override and profile == "REMOTE_ARM_OVERRIDE" and not self.fresh(now_s):
            return "operator_override_active"
        return ""
```

- [ ] **Step 3: nonblocking flock helper 구현**

```python
import fcntl
import os

class CanOwnershipError(RuntimeError):
    pass

class CanOwnerLock:
    def __init__(self, path="/run/powertrain/can0.lock"):
        self.path = path
        self.fd = None

    def acquire(self):
        self.fd = os.open(self.path, os.O_RDWR | os.O_CREAT, 0o640)
        try:
            fcntl.flock(self.fd, fcntl.LOCK_EX | fcntl.LOCK_NB)
        except BlockingIOError as exc:
            os.close(self.fd)
            self.fd = None
            raise CanOwnershipError(f"CAN already owned: {self.path}") from exc

    def close(self):
        if self.fd is not None:
            os.close(self.fd)
            self.fd = None
```

잠금 파일은 stale 여부와 무관하게 삭제하지 않는다. 커널 lock은 fd 종료로 해제된다.

`CanOwnerLock`은 내부 primitive이고 entry point가 사용하는 공개 API는 Interfaces에 선언한
`RealCanSession` 하나다: context manager로 구현해 진입 시 lock을 획득하고 종료 시 bus 다음 fd
순서로 해제하며, 획득 시각·PID·프로세스명·lock 경로를 immutable owner snapshot 속성으로

경로의 fd 누수를 막는다.

- Produces: `/teleop/cmd_vel` 또는 `/arm/teleop_jog` 중 하나, gateway mode ACK와 channel health.
- Produces: `chassis_node` 내부 final `DriveCommand`, `WheelStopPredicate.update(...) -> bool`과
  zero-confirmed owner transition. 외부 ROS `/cmd_vel` writer나 subscriber를 만들지 않는다.

Remote-input wire는 newline-delimited JSON schema v1로 고정한다. 각 record는 2 KiB 이하이며
`schema_version`, random `session_id`, 단조 `sequence`, `client_monotonic_ns`, requested `mode`,
`deadman`, normalized axes, D-pad, mode chord와 E-stop edge를 가진다. `client_monotonic_ns`는
correlation·로그용이고 freshness는 Jetson local receive monotonic age로 판정한다. 같은 session의
중복·역행 sequence, 미인식 version/mode, NaN·범위 밖 axis, oversize·부분 record는 출력 없이
`CONTRACT_VIOLATION`으로 기록한다. 새 TCP connection은 새 session ID를 요구하며 이전 session의
마지막 frame이나 mode를 복원하지 않는다.

- [ ] **Step 1: 중첩·무정지 handover 실패시험 작성**

자율 nonzero 상태에서 teleop 선택 시 즉시 source를 바꾸지 않고 0 command와 wheel-stop ACK를
거쳐야 한다. 두 입력이 동시에 fresh해도 선택되지 않은 source는 출력에 영향을 주지 않으며,
source stale은 motion hold를 만들어야 한다.

remote-input fixture는 30 Hz 정상 frame 뒤 packet 정지, TCP reconnect, session 교체, sequence
중복·역행, malformed/oversize record를 주입한다. `DRIVE→ARM` 요청 중에는 drive를 먼저 0으로 만들고
qualified wheel-stop과 arm stationary ACK 전까지 `/arm/teleop_jog`를 0으로 유지해야 한다.
`ARM→DRIVE` 요청 중에는 arm jog와 gripper를 먼저 0으로 만들고 `STOW_REQUEST` 뒤 fresh
`STOWED_LOCKED` 전까지 `/teleop/cmd_vel`을 0으로 유지해야 한다. 어떤 fixture에서도 두 output이
동시에 nonzero이면 실패다.

wheel-stop fixture는 여섯 wheel별 threshold map이 모두 존재하고 `qualified=true`일 때만 production
판정을 허용한다. sample은 header가 fresh·단조이고 정확히 6개 unique wheel, finite 속도,
`healthy=true`, 모든 drive/steer stale=false, axis/fault=0이며 authority final output이 0이어야 한다.
어느 조건이든 깨지면 dwell을 리셋한다. threshold는
`max(전진·후진·좌/우 pivot 뒤 측정 noise 상한×1.5, encoder_resolution_floor)`로 정하며 측정 전
exact-zero floor를 승인하지 않는다. `median-of-6`은 단일 회전 wheel을 숨기므로 금지한다.

- [ ] **Step 2: 순수 authority FSM 구현**

`CommandAuthority`는 별도 ROS node가 아니라 순수 코어로 만들고 `chassis_node`가 단독 소유한다.
상태는 `IDLE, AUTONOMY, STOPPING_FOR_HANDOVER, TELEOP, MOTION_HOLD`로 제한한다. source 전환은
`STOPPING_FOR_HANDOVER`에서 0을 출력하고 qualified per-wheel `WheelStopPredicate`가 설정 dwell 동안
연속 true인 뒤 완료한다. 개발 기본 `wheel_stop.yaml`은 `qualified: false`, 빈 threshold map,
`dwell_ms: 300`으로 두어 HIL 전 production handover와 arm work permit이 명시적으로 실패하게 한다.
여기서 실패는 대기가 아니라 **즉시 거부**다: `wheel_stop.yaml`이 unqualified이면 두 FSM은
`STOPPING_FOR_*`에 진입하지 않고 전환 요청을 거부해 현재 상태를 유지하며 거부 사유를 운영자에게
통보한다. 또한 `STOPPING_FOR_HANDOVER`·`STOPPING_FOR_ARM`·`STOPPING_FOR_DRIVE`에는 설정 가능한
timeout을 두고 만료 시 `MOTION_HOLD`로 전이한다 — 어떤 STOPPING 상태도 무한 대기하지 않으며,
`MOTION_HOLD` 진입 후 다음 전환 요청은 운영자 확인 뒤에만 수락한다. Step 1 fixture에 두 경우
(unqualified 즉시 거부, STOPPING timeout→MOTION_HOLD)를 포함한다.

`RemoteInputGateway`의 state는 `DISCONNECTED, DRIVE, STOPPING_FOR_ARM, ARM,
STOPPING_FOR_DRIVE, MOTION_HOLD`로 둔다. client의 mode 표시는 요청일 뿐이며 Jetson state ACK가
권위다. initial input timeout은 0.20 s로 시작하고 qualification에서 loss·jitter 분포를 측정해
고정한다. timeout·deadman release·controller feedback stale은 다음 tick에 두 출력을 0으로 만들고
자동 재연결은 허용하되 이전 nonzero command와 mode는 자동 복원하지 않는다.

- [ ] **Step 3: 입력 adapter 구현**

노트북 `remote_operation_client.py`는 현재 `laptop_client_chassis.py`의 reconnect·30 Hz raw-input
경험을 계승하되 schema v1 frame만 보낸다. wire semantic은 requested mode, deadman, joint select,
signed jog, gripper intent와 E-stop edge로 고정한다. 아래 물리 키매핑은 **초기 후보**이며 HIL과
운전자 피드백 뒤 SDL GUID별 versioned config로 변경할 수 있다.

| 초기 후보 입력 | DRIVE | ARM |
|---|---|---|
| R2 / L2 | 전진 / 후진 | gripper open / close |
| 좌스틱 X | yaw | 0 |
| 우스틱 Y | 0 | selected joint signed jog |
| D-pad 좌 / 우 | 0 | `joint_1`~`joint_5` 순환 선택 |
| L1 hold | deadman | deadman |
| CREATE+OPTIONS 1 s hold | ARM 요청 | DRIVE+stow 요청 |
| ○ edge | manual latched E-stop | manual latched E-stop |

Jetson gateway만 semantic input을 `/teleop/cmd_vel` 또는 한 joint 이름·signed velocity를 가진
`/arm/teleop_jog`로 변환한다. ARM mode에서 두 trigger 동시 입력은 gripper hold, invalid selected
joint와 비선형/NaN axis는 전체 arm hold다. L1을 놓으면 현재 자세 hold이며 all-zero `home` command는
초기 production UI에 노출하지 않는다.

DualSense와 web remote는 모터 클래스를 import하지 않는다. legacy direct-CAN 모드는 별도 명시적
CLI flag와 운영자 팔 접힘 확인을 요구하는 진단 경로로 격리하고 production launch에는 포함하지
않는다.

production adapter는 `powertrain_control` 서비스의 ROS image에서 실행한다. `Dockerfile.ros`에 승인된
pygame/SDL 또는 web-input dependency를 설치하고 headless device 접근을 smoke-test한다.
`/arm/teleop_jog`의 `control_msgs/msg/JointJog` 사용을 위해 `package.xml`에 `control_msgs`
exec_depend를 추가하고 `Dockerfile.ros`에 `ros-humble-control-msgs`를 설치한다. pygame가 있는

- [ ] **Step 1: FSM 실패시험 작성**

다음을 독립 시험으로 고정한다.

- operator override 요청 시 active mission을 abort하고 `ArrivalStatus` 재발행을 먼저 중단한다.

- [ ] **Step 2: 영속 ID allocator 구현**

설치 스크립트가 root:root 0750 `/var/lib/powertrain`을 만들고 Compose가 같은 경로로 bind한다.

0 이하 또는 `2_147_483_647` 도달은 새 작업을 거부하고 motion hold한다.

- [ ] **Step 3: 순수 FSM 구현**

상태는 `READY, DRIVE, STOP_REQUESTED, EVENT_HOLD, ARM_WORK, STOW_VERIFY, RESUME,
COMPLETE, FAILED_HOLD, GRIP_LOST_HOLD`로 제한한다. timeout은 상태를 자동 건너뛰지 않고
hold와 운영자 통보만 만든다.

`FAILED_HOLD`에는 wire status 원문, `mission_id`, 수신 stamp, 마지막 locked posture와 팔 측
latched 여부를 보존한다. 내부 실패는 motion hold로 시작하고 작업 종류에 맞는 fresh locked posture,
필요한 팔 fault reset과 supervisor의 명시적 skip/retry 전이가 모두 확인된 뒤에만 빠져나온다.
픽업 실패는 release 확인 뒤 `STOWED_LOCKED`로 종결한다. drop 실패 또는 release 불확실은
`CARRYING_LOCKED`나 `GRIP_UNCERTAIN/FAILED_HOLD`를 유지하며, release 확인 없이 빈 적재
`STOWED_LOCKED`로 축약하지 않는다.

`GRIP_LOST_HOLD`의 이탈 전이는 둘뿐이다: ① bounded regrasp 성공이 동일 `mission_id`의 fresh
`CARRYING_LOCKED`로 확인된 뒤 supervisor의 명시적 전이, ② 운영자 확인·승인 →
`ArmInterlock.clear_grip_lost(authorized=True)` → 작업 종류에 맞는 fresh locked posture 확인 →
supervisor의 명시적 전이. 새 locked heartbeat 단독·timeout·팔 재시작은 이탈을 만들지 않는다.
Step 1 시험에 두 이탈 경로의 양성 시험과 "heartbeat만으로 이탈 없음" 음성 시험을 함께 둔다.

`MissionSupervisor`는 별도 ROS process가 아니라 ROS 없는 순수 코어로 만들고 `chassis_node`가
소유한다. 따라서 Task 2의 mode-intent 경계를 직접 호출하며 내부용 ROS 토픽이나 두 번째
`/chassis_mode` writer를 만들지 않는다. 예외는 chassis tick 밖에서 포착해 mission hold로 변환하고
50 Hz safety loop를 종료시키지 않는다.

  판정과 보수적 운반 profile을 적용한다. 2026-07-14 코드 재검증 기준 잔여 deliverable:
  ① payload-aware `STOW_REQUEST`(운반 중 release 금지, `CARRYING_LOCKED`로 접힘),
  ② locked 판정의 approved-pose proximity 검사(정지 안정성만으로는 불충분)와 실제 stow 모션,
  ③ controller fault·torque hold(2단계). ①②는 합동 실차 주행 전 필수, ③은 후속.
  내부 실패 8종은 선택 산출물이며 base gate 선행조건이 아니다. 원격 팔 조종을 위해 D435i 단일
  owner의 raw RGB 848×480×30 SRT `:5002`, latest-only YOLO와 UDP metadata `:5003`, sender와
  receiver의 frame age/sequence feedback, deadman 연동을 제공한다. L515 `:5000`과 동시 실행해도
  두 receiver가 각각 29 fps 이상이어야 한다.
- Requires robot-arm remote-control deliverable: `joint_1`~`joint_5`가 포함된 최종 URDF,
  controller와 fresh `/joint_states`, `control_msgs/msg/JointJog` 입력, MoveIt Servo 5축 config,
  gripper `FollowJointTrajectory`, controller fault와 joint-stop ACK를 제공한다. 예상 수정 경계는
  `dynamixel_control/arm_command_authority.py`, `arm_fsm_node.py`, `moveit_dynamixel_bridge.py`,
  `robot_arm_description/config/controllers.yaml`, `robot_arm_moveit_config/config/ros2_controllers.yaml`과
  새 Servo config다. 실제 파일 소유권과 커밋은 로봇팔 팀에 있다.
- `ArmCommandAuthority`는 `AUTO_FSM, STOPPING_FOR_REMOTE, REMOTE_JOG, STOPPING_FOR_AUTO, HOLD`를
  단독 소유한다. REMOTE 진입 전에 active MoveIt action cancel ACK와 모든 joint 정지를 확인하며,
  REMOTE 중에는 FSM trajectory와 arrival acceptance를 막는다. 원격 `JointJog`만 Servo의
  joint-limit·collision·singularity scaling을 거쳐 단일 Dynamixel bridge에 도달한다. gripper도 같은
  authority가 소유하고 두 trigger 동시 입력은 hold다.
- 원격 조작 중 wire status는 기존 `EXECUTING`을 10 Hz로 사용한다. 새 `ArmStatus` 어휘나 msg를
  추가하지 않는다. DRIVE 복귀는 remote jog/gripper 0 → `STOW_REQUEST` → `STOWING` → fresh
  `STOWED_LOCKED` 순서이며, all-zero `home` 단축키는 production에 노출하지 않는다.

- [ ] **Step 1: 양 팀 동기 계약시험**

vendored msg drift 0을 확인하고 문자열 fixture를 양쪽 노드에 재생한다. 먼저
`contract_v2_verified=false`에서 새 chassis가 `CORNERING` compatibility lock만 발행하고
`DRIVING/MISSION_STOP/ArrivalStatus`를 0회 발행하는지 확인한다. 구 팔은 LOCKED에서 이전 모션을
재개하지 않고 차체 nonzero도 0이어야 한다. 이 검증은 legacy와 production을 혼용해도 된다는 뜻이
아니며, v2 양 팀 배포가 끝날 때까지 실물 합동 arm을 금지하는 deployment gate다.

arm 직전 preflight는 양 checkout의 msg 5종 SHA-256 일치, 네 계약 토픽 endpoint 존재와 type/QoS,
`ROS_DOMAIN_ID=0`, 실차 `use_sim_time=false`, 합의한 RMW 통신을 확인한다. 시뮬레이션에서는 mock을
포함한 모든 node가 `use_sim_time=true`와 같은 `/clock`을 사용한다. `DetectedObjectArray`는
Best Effort/Keep Last 1, arm/chassis heartbeat는 Reliable/Keep Last 1, arrival event는
Reliable/Keep Last 10을 요구한다.

- [ ] **Step 2: software-only fault injection**

heartbeat kill, DDS 재연결 stale burst, stamp 역행, 두 토픽 순서 역전, duplicate arrival,
delayed DONE, arm restart, FAILED, GRIP_LOST와 override를 순서대로 주입한다. override는 active
mission 취소와 arrival 재발행 중단 전에는 거부되고, GRIP_LOST 중에는 항상 거부되며, 팔 heartbeat
복귀 즉시 drive permission이 0이 되는지 확인한다. 미인식 status는 즉시 hold하고
`CONTRACT_VIOLATION` 진단을 남긴다.

D435i fake capture는 raw sender와 인위적으로 느린 YOLO worker에 동시에 frame을 공급한다. YOLO
backlog 없이 최신 frame으로 건너뛰고 raw receiver가 첫 완전한 5초 window에서 29 fps 이상인지,
metadata loss·reorder·stale에서 overlay만 사라지는지 확인한다. D435i raw receiver feedback 또는
operator deadman이 stale하면 원격 팔 명령은 hold되고 차체는 `MISSION_STOP`을 유지해야 한다.
overlay metadata stale만으로 latched E-stop이나 chassis motion을 만들지 않는다.

DualSense fixture는 schema version/session/sequence, 30 Hz frame과 versioned mapping config의
deadman, joint selection, signed jog, gripper, mode request를 재생한다. 초기 후보 config는
L1·D-pad·우스틱·R2/L2·CREATE+OPTIONS를 사용한다. client mode가 먼저
바뀌어도 Jetson ACK 전 출력 의미가 바뀌지 않고, DRIVE/ARM nonzero overlap이 0인지 확인한다.
ARM 진입 중 wheel/joint stop 또는 FSM cancel ACK 하나를 지연시키면 jog 0을 유지해야 한다. remote
input·D435i feedback·`/joint_states` 중 하나를 kill하면 다음 arm control tick에 hold하고, reconnect한
이전 session frame은 재생하지 않는다.

로봇팔 authority fixture는 AUTO trajectory 실행 중 REMOTE 요청, REMOTE 중 arrival 재발행, stale
joint feedback, controller fault, joint limit, self/base collision과 singularity 접근을 주입한다. 모든
경우 단일 low-level writer와 bounded stop을 보장하고 direct `/dynamixel/goal_position` publisher가
존재하면 production preflight를 실패시킨다.

그리퍼 effort detector는 무부하·정상 파지·진동·강제 미끄러짐을 반복해 miss와 false positive를
기록한다. 검출기가 qualification을 통과하지 못하면 보고서에 `GRIP_LOST unsupported`를 명시하고
최저속 운반·하역 전 재확인 fallback을 시험한다.

- [ ] **Step 3: 바퀴 부양 합동 HIL**

선행조건은 Observability Task 6의 dual receiver·feedback와 operator console 완료다. 그 전까지
Task 4 gateway의 ARM output은 false를 유지하며, 이 HIL에서만 production enable 후보를 검증한다.

팔 합동 전 ODrive 11~16 각각을 전진·후진·좌 pivot·우 pivot 뒤 정지시키는 조건을 10회씩 반복하고
각 정지 구간 5초를 50 Hz로 기록한다. 각 wheel threshold 후보는
`max(관측 최대 정지 노이즈 절댓값×1.5, encoder_resolution_floor)`로 계산한다. 0.05 rev/s는 초기
acceptance 후보일 뿐 측정 없이 production 상수로 동결하지 않는다. 요구 제동·작업 정지 성능을
만족할 수 없는 threshold가 나오면 해당 축 qualification을 실패시킨다. 이때 median-of-6이나 문제 축
무시는 금지하고 node 12/16 HALL 접지·필터 보강 또는 encoder 문제를 해결한 뒤 재측정한다.

같은 HIL에서 L515 1280×720×30과 D435i 848×480×30, 로봇팔 YOLO, MoveIt/FSM, chassis 50 Hz를
30분 동시에 실행한다. 같은 완전한 5초 window에서 두 receiver가 각각 29 fps 이상, sender drop과
고아 encoder 0, YOLO backlog 0, OOM 0이어야 한다. D435i 분리·재연결은 L515 Gateway나 chassis를
재시작하지 않고, L515 receiver 단절은 D435i owner를 죽이지 않아야 한다.

qualification 대상 mapping config로 DRIVE→ARM을 전환해 `joint_1`~`joint_5`를 각각 양방향 저속
jog하고 gripper open/close를 각각 시험한다. 매 관절에서 deadman release, 입력 TCP 단절과 D435i
receiver feedback 단절 시 bounded
hold를 측정한다. 자동 FSM trajectory 실행 중 ARM 요청은 cancel·joint-stop ACK 전까지 거부되어야
하며, ARM→DRIVE는 `STOWED_LOCKED` 전까지 ARM input이 차체 속도로 재해석되지 않아야 한다. 30분 동안
FSM/Servo/bridge 중복 writer, stale command replay와 chassis/arm nonzero overlap은 모두 0이어야 한다.
node 13/14가 포함되지 않은 측정으로 production YAML을 qualified로 바꾸지 않는다. 합격한 여섯
threshold와 300 ms 이상 dwell을 `wheel_stop.yaml`에 기록하고, 의도적으로 한 wheel만 움직이는 시험이
work permit을 만들지 않는지 확인한다.

실제 wheel 정지, work permit, pickup+자동 stow, CARRYING_LOCKED, drop+자동 stow,
STOWED_LOCKED와 resume을 한 사이클로 기록한다. 실패·skip에는 별도로 STOW_REQUEST를 주입한다.
이 단계 전에는 합동 arm 금지를 해제하지 않는다.
